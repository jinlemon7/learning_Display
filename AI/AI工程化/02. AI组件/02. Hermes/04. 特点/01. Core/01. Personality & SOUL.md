Hermes Agent 的个性完全可自定义 SOUL.md 是主要身份标识 ——它是系统提示中的第一个信息，定义了agent的身份。  

- SOUL.md — 一个持久化的角色文件，位于 HERMES_HOME 目录中，用作agent的身份标识（系统提示符中的第一个位置）。
- 内置或自定义 /personality 预设 —— 会话级系统提示叠加层

如果你想改变 Hermes 的身份——或者用完全不同的agent形象替换它——编辑 SOUL.md  

# SOUL.md 目前的工作原理
  Hermes 现在会自动在以下位置初始化一个默认的 SOUL.md 文件：  
  ```bash
  ~/.hermes/SOUL.md
  ```

  更准确地说，它使用当前实例的 HERMES_HOME ，因此如果您使用自定义主目录运行 Hermes，它将使用：  
  ```bash
  $HERMES_HOME/SOUL.md
  ```

## 重要行为
  - SOUL.md 是agent的主要身份标识。 它占据系统提示符中的第一个位置，替换了硬编码的默认身份标识。
  - 如果初始 SOUL.md 文件尚不存在，Hermes 会自动创建一个。
  - 已存在的用户 SOUL.md 文件永远不会被覆盖。
  - Hermes 仅从 HERMES_HOME 加载 SOUL.md
  - Hermes 不会在当前工作目录中查找 SOUL.md 文件。
  - 如果 SOUL.md 存在但为空或无法加载，Hermes 将回退到内置的默认身份。
  - 如果 SOUL.md 包含内容，则在安全扫描和截断后，该内容将原封不动地注入。
  - SOUL.md 在上下文文件部分中没有重复出现——它只出现一次，作为标识。

  这使得 SOUL.md 成为真正的基于用户或实例的身份标识，而不仅仅是一个附加层。  

# 为什么选择这种设计
  这使得人物性格具有可预测性。  

  如果 Hermes 从你启动它时所在的任意目录加载 SOUL.md ，那么你的角色配置可能会在不同项目之间发生意外变化。而如果仅从 HERMES_HOME 目录加载，则角色配置将只属于 Hermes 实例本身。  

  这也使得用户更容易接受培训：  
  - “编辑 ~/.hermes/SOUL.md 以更改 Hermes 的默认性格。”

# 在哪里编辑
  For most users:  对于大多数用户而言：  
  ```bash
  ~/.hermes/SOUL.md
  ```

  如果您使用自定义用户主目录：  
  ```bash
  $HERMES_HOME/SOUL.md
  ```

# SOUL.md 文件中应该包含哪些内容？  
  可将其用于持久的声音和个性塑造，例如：  
  - tone  语气
  - communication style  沟通风格
  - level of directness  直接程度
  - default interaction style  默认交互方式
  - what to avoid stylistically  风格上应该避免什么
  - how Hermes should handle uncertainty, disagreement, or ambiguity  Hermes 应该如何处理不确定性、分歧或歧义

  减少用于：  
  - one-off project instructions  一次性项目说明
  - file paths  文件路径
  - repo conventions  仓库约定
  - temporary workflow details  临时工作流程详情

  这些内容应该放在 AGENTS.md 中，而不是 SOUL.md 中。  

# Good SOUL.md 内容
  一份好的 SOUL 文件是：  
  - stable across contexts  在不同环境下保持稳定
  - broad enough to apply in many conversations  适用范围很广，可以用于许多对话
  - specific enough to materially shape the voice  具体到足以实质性地塑造声音
  - focused on communication and identity, not task-specific instructions  侧重于沟通和身份认同，而非具体任务指令

## 例子
  ```bash
  # Personality
  
  You are a pragmatic senior engineer with strong taste.
  You optimize for truth, clarity, and usefulness over politeness theater.
  
  ## Style
  - Be direct without being cold
  - Prefer substance over filler
  - Push back when something is a bad idea
  - Admit uncertainty plainly
  - Keep explanations compact unless depth is useful
  
  ## What to avoid
  - Sycophancy
  - Hype language
  - Repeating the user's framing if it's wrong
  - Overexplaining obvious things
  
  ## Technical posture
  - Prefer simple systems over clever systems
  - Care about operational reality, not idealized architecture
  - Treat edge cases as part of the design, not cleanup
  ```

# Hermes 在提示中注入了什么
  SOUL.md 内容直接放入系统提示符的第一个插槽——agent身份位置。没有添加任何包装语言。  

  内容经过以下流程：  
  1. prompt-injection scanning  即时注射扫描
  2. truncation if it is too large  如果过大则截断

  如果文件为空、仅包含空格或无法读取，Hermes 将回退到内置的默认身份（“您是 Hermes Agent，由 Nous Research 开发的智能 AI 助手……”）。当设置了 skip_context_files 时（例如，在子agent/委托上下文中），此回退机制也适用。  

# 安全扫描
  SOUL.md 与其他包含上下文的文件一样，在包含之前会进行扫描，以查找提示注入模式。  

  这意味着你仍然应该专注于角色/声音，而不是试图偷偷加入奇怪的元指令。  

# SOUL.md 与 AGENTS.md
  这是最重要的区别。  

## SOUL.md
   用于：  
   - identity  身份
   - tone  语气
   - style  风格
   - communication defaults  通信默认设置
   - personality-level behavior  人格层面的行为

## AGENTS.md
  用于：  
  - project architecture  项目架构
  - coding conventions  编码规范
  - tool preferences  工具偏好设置
  - repo-specific workflows  仓库特定工作流程
  - commands, ports, paths, deployment notes  命令、端口、路径、部署说明

  一条有用的规则：  
  - 如果它要如影随形，那它就属于 SOUL.md
  - 如果它属于某个项目，则应放在 AGENTS.md 中。

# SOUL.md 与 /personality
  SOUL.md 是你持久的默认人格。  
  /personality 是一个会话级别的覆盖层，用于更改或补充当前的系统提示。  

  所以：  
  - SOUL.md = 基线语音
  - /personality = 临时模式切换

  例如：  
  - 保持务实的默认灵魂状态，然后使用 /personality teacher 进行辅导对话。
  - 保持简洁的灵魂，然后使用 /personality creative 进行头脑风暴。

# 内置个性
  Hermes 内置了多种人格，您可以使用 /personality 切换到这些人格。  
  | Name | Description |
  |------|-------------|
  | helpful | Friendly, general-purpose assistant / 友善的通用助理 |
  | concise | Brief, to-the-point responses / 简明扼要的回答 |
  | technical | Detailed, accurate technical expert / 细致、准确的技术专家 |
  | creative | Innovative, outside-the-box thinking / 创新、跳出固有思维模式 |
  | teacher | Patient educator with clear examples / 细心教育者提供清晰的案例 |
  | kawaii | Cute expressions, sparkles, and enthusiasm ★ / 可爱的表情、闪亮的光芒和热情 ★ |
  | catgirl | Neko-chan with cat-like expressions, nya~ / 猫咪表情的 Neko-chan，喵~ |
  | pirate | Captain Hermes, tech-savvy buccaneer / Hermes 船长，精通科技的海盗 |
  | shakespeare | Bardic prose with dramatic flair / 充满戏剧张力的吟游诗人式散文 |
  | surfer | Totally chill bro vibes / 完全放松的兄弟氛围 |

# 通过指令切换人格

## CLI  命令行界面
  ```bash
  /personality
  /personality concise
  /personality technical
  ```

## 即时通讯平台
  ```bash
  /personality teacher
  ```
  
  这些只是方便的叠加层，但除非叠加层对 Hermes 的默认个性进行实质性的改变，否则全局 SOUL.md 文件仍然会赋予 Hermes 其持久的默认个性。  

# 配置中的自定义个性
  您还可以在 ~/.hermes/config.yaml 的 agent.personalities 下定义命名的自定义个性。  

  ```bash
  agent:
    personalities:
      codereviewer: >
        You are a meticulous code reviewer. Identify bugs, security issues,
        performance concerns, and unclear design choices. Be precise and constructive.
  ```

  然后切换到它：  
  ```bash
  /personality codereviewer
  ```

# 推荐工作流程
  一个可靠的默认设置是：  
  1. 在 ~/.hermes/SOUL.md 中保留一份深思熟虑的全局 SOUL.md
  2. 将项目说明放入 AGENTS.md 中
  3. 仅在需要临时切换模式时使用 /personality

  这样你就得到了：  
  - a stable voice  稳定的声音
  - project-specific behavior where it belongs  项目特定行为（如果它属于这个范畴）
  - temporary control when needed  必要时采取临时控制措施

# 个性如何与完整提示相互作用
  从宏观层面来看，提示堆栈包括：  
  1. SOUL.md （agent身份——如果 SOUL.md 不可用，则使用内置的备用身份）
  2. 工具感知行为指导
  3. 记忆/用户上下文
  4. 技能指导
  5. 上下文文件（ AGENTS.md 、 .cursorrules ）
  6. 时间戳
  7. 平台特定的格式提示
  8. 可选的系统提示符覆盖层，例如 /personality

  SOUL.md 是基础——其他一切都建立在其之上。
