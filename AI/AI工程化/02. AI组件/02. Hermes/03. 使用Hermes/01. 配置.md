所有设置都存储在 ~/.hermes/ 目录中，以便于访问。  

# 目录结构
  ```text
  ~/.hermes/
  ├── config.yaml     # Settings (model, terminal, TTS, compression, etc.)
  ├── .env            # API keys and secrets
  ├── auth.json       # OAuth provider credentials (Nous Portal, etc.)
  ├── SOUL.md         # Primary agent identity (slot #1 in system prompt)
  ├── memories/       # Persistent memory (MEMORY.md, USER.md)
    ├── MEMORY.md
    ├── USER.md
  ├── skills/         # Agent-created skills (managed via skill_manage tool)
  ├── cron/           # Scheduled jobs
  ├── sessions/       # Gateway sessions
  └── logs/           # Logs (errors.log, gateway.log — secrets auto-redacted)
  ```

# 管理配置
  ```bash
  hermes config              # View current configuration
  hermes config edit         # Open config.yaml in your editor
  hermes config set KEY VAL  # Set a specific value
  hermes config check        # Check for missing options (after updates)
  hermes config migrate      # Interactively add missing options
  
  # Examples:
  hermes config set model anthropic/claude-opus-4
  hermes config set terminal.backend docker
  hermes config set OPENROUTER_API_KEY sk-or-...  # Saves to .env
  ```

# 配置优先级
  设置按以下顺序解析（优先级最高优先）：   
  1. CLI 参数 — 例如， hermes chat --model anthropic/claude-sonnet-4 （每次调用覆盖）
  2. ~/.hermes/config.yaml — 所有非机密设置的主要配置文件
  3. ~/.hermes/.env — 环境变量的备用文件；密钥（API 密钥、令牌、密码） 必需 。
  4. 内置默认值 ——当没有其他设置时，硬编码的安全默认值

  > [!WARNING]  
  > 密钥（API 密钥、机器人令牌、密码）放在 .env 中。  
  > 其他所有内容（模型、终端后端、压缩设置、内存限制、工具集）都放在 config.yaml 中。  
  > 如果两个文件都设置了，则非密钥设置以 config.yaml 文件为准。   

# 环境变量替代
  您可以使用 ${VAR_NAME} 语法在 config.yaml 中引用环境变量：  
  ```bash
  auxiliary:
    vision:
      api_key: ${GOOGLE_API_KEY}
      base_url: ${CUSTOM_VISION_URL}
  
  delegation:
    api_key: ${DELEGATION_KEY}
  ```
  - 单个值中可以包含多个引用： url: "${HOST}:${PORT}" 。如果引用的变量未设置，则占位符将保持不变（ ${UNDEFINED_VAR} 保持不变）。仅支持 ${VAR} 语法——不会展开纯 $VAR 。

## 提供者超时
  您可以设置 providers.`<id>`.request_timeout_seconds 来设置全局请求超时时间，还可以 providers.`<id>`.models.`<model>`.timeout_seconds 来设置特定于模型的超时时间。此设置适用于所有传输方式（OpenAI-wire、原生 Anthropic、Anthropic 兼容）上的主 Turn 客户端、回退链、凭证轮换后的重建，以及（对于 OpenAI-wire）每个请求的超时时间参数——因此，配置的值会覆盖旧的 HERMES_API_TIMEOUT 环境变量。  

  您还可以为非流式过期调用检测器设置 providers.`<id>`.stale_timeout_seconds ，并为特定模型覆盖设置 providers.`<id>`.models.`<model>`.stale_timeout_seconds 这将取代旧的 HERMES_API_CALL_STALE_TIMEOUT 环境变量。  

  如果不设置这些参数，则会保留旧版默认值（ HERMES_API_TIMEOUT=1800 秒， HERMES_API_CALL_STALE_TIMEOUT=300 秒，Anthropic 原生值为 900 秒）。目前尚未针对 AWS Bedrock 进行配置（ bedrock_converse 和 AnthropicBedrock SDK 路径均使用 boto3，并带有其自身的超时配置）。  

# 终端后端配置
  Hermes 支持七个终端后端。每个后端都决定了代理的 shell 命令实际执行的位置——本地计算机、Docker 容器、通过 SSH 连接的远程服务器、Modal 云沙箱（直接或通过 Nous 管理的网关）、Daytona 工作区、Vercel 沙箱或 Singularity/Apptainer 容器。  
  ```bash
  terminal:
    backend: local    # local | docker | ssh | modal | daytona | vercel_sandbox | singularity
    cwd: "."          # Gateway/cron working directory (CLI always uses launch dir)
    timeout: 180      # Per-command timeout in seconds
    env_passthrough: []  # Env var names to forward to sandboxed execution (terminal + execute_code)
    singularity_image: "docker://nikolaik/python-nodejs:python3.11-nodejs20"  # Container image for Singularity backend
    modal_image: "nikolaik/python-nodejs:python3.11-nodejs20"                 # Container image for Modal backend
    daytona_image: "nikolaik/python-nodejs:python3.11-nodejs20"               # Container image for Daytona backend
  ```

## 后端概览
  | Backend | 命令运行位置 | 隔离 | 最适合 |
  |---------|--------------|------|--------|
  | local | 您的机器直接 | 无 | 开发，个人用途 |
  | docker | 单个持久性 Docker 容器（在会话、/new、子代理之间共享） | 完整（命名空间，cap-drop） | 安全沙箱、CI/CD |
  | ssh | 通过 SSH 远程服务器 | 网络边界 | 远程开发，强大的硬件 |
  | modal | Modal 云沙箱 | 完整（云虚拟机） | 临时云计算评估 |
  | daytona | Daytona 工作空间 | 完整（云容器） | 托管云开发环境 |
  | vercel_sandbox | Vercel Sandbox | 完整（云端微型虚拟机） | 基于快照的文件系统持久化的云执行 |
  | singularity | Singularity/Apptainer 容器 | 命名空间（--containall） | 高性能计算集群，共享机器 |

## local 后端
  默认设置。命令直接在您的计算机上运行，​​无需隔离。无需特殊设置。 
  ```bash
  terminal:
    backend: local
  ```

  > [!WARNING]
  > agent 程序拥有与您的用户帐户相同的文件系统访问权限。您可以使用 hermes tools 禁用不需要的工具，或者切换到 Docker 进行沙箱隔离。

## Docker 后端
  在 Docker 容器内运行命令，并进行安全加固（放弃所有功能，禁止权限提升，限制 PID）。  

  Hermes 采用单一持久容器，而非每个命令一个容器。 首次使用时，Hermes 会启动一个长期运行的容器，并将所有终端、文件和 execute_code 调用通过 docker exec 路由到该容器内——无论会话、 /new 、 /reset 还是 delegate_task 子代理——在 Hermes 进程的整个生命周期内都是如此。工作目录的更改、已安装的软件包以及 /workspace 中的文件都会像本地 shell 一样，在每次工具调用之间传递。关闭时，容器也会停止并被移除。详情请参阅下方的容器生命周期 。  

  ```bash
  terminal:
    backend: docker
    docker_image: "nikolaik/python-nodejs:python3.11-nodejs20"
    docker_mount_cwd_to_workspace: false  # Mount launch dir into /workspace
    docker_run_as_host_user: false   # See "Running container as host user" below
    docker_forward_env:              # Env vars to forward into container
      - "GITHUB_TOKEN"
    docker_volumes:                  # Host directory mounts
      - "/home/user/projects:/workspace/projects"
      - "/home/user/data:/data:ro"   # :ro for read-only
  
    # Resource limits
    container_cpu: 1                 # CPU cores (0 = unlimited)
    container_memory: 5120           # MB (0 = unlimited)
    container_disk: 51200            # MB (requires overlay2 on XFS+pquota)
    container_persistent: true       # Persist /workspace and /root across sessions
  ```

  要求： 已安装并运行 Docker Desktop 或 Docker Engine。Hermes 会探测 $PATH 变量以及常见的 macOS 安装位置（例如 /usr/local/bin/docker 、 /opt/homebrew/bin/docker 和 Docker Desktop 应用包）。Podman 已默认支持：如果两者都已安装，则设置 HERMES_DOCKER_BINARY=podman （或完整路径）即可强制使用 Podman。  

  容器生命周期： Hermes 会为所有终端和文件工具调用（跨会话、 /new 、 /reset 和 delegate_task 子代理）在 Hermes 进程的整个生命周期内重用一个长期运行的容器（` docker run -d ... sleep 2h ）。命令通过登录 shell 使用 docker exec 运行，因此工作目录的更改、已安装的软件包以及 /workspace 中的文件都会在每次工具调用之间保留。容器会在 Hermes 关闭时（或空闲清理程序回收时）停止并移除。  

  通过 delegate_task(tasks=[...]) ` 生成的并行子代理共享同一个容器——并发的 cd 、环境变量变更以及对同一路径的写入操作会发生冲突。如果子代理需要一个隔离的沙箱，则必须通过 register_task_env_overrides() 注册一个针对每个任务的镜像覆盖，而强化学习和基准测试环境（例如 TerminalBench2、HermesSweEnv 等）会自动为其每个任务的 Docker 镜像执行此操作。  

  Security hardening:  安全加固：  
  - --cap-drop ALL ，仅添加 DAC_OVERRIDE 、 CHOWN 和 FOWNER 参数
  - --security-opt no-new-privileges
  - --pids-limit 256
  - 为 /tmp (512MB)、 /var/tmp (256MB)、 /run (64MB) 设置了大小受限的 tmpfs 文件系统

  凭证转发： docker_forward_env 中列出的环境变量首先从您的 shell 环境解析，然后从 ~/.hermes/.env 解析。技能还可以声明 required_environment_variables ，这些变量会自动合并。  

## SSH 后端
  通过 SSH 在远程服务器上运行命令。使用 ControlMaster 实现连接重用（5 分钟空闲保持连接）。默认启用持久 shell——状态（当前工作目录、环境变量）在命令执行后仍然保留。  
  ```bash
  terminal:
    backend: ssh
    persistent_shell: true           # Keep a long-lived bash session (default: true)
  ```

  必需的环境变量  
  ```bash
  TERMINAL_SSH_HOST=my-server.example.com
  TERMINAL_SSH_USER=ubuntu
  ```

  可选的：  
  | Variable | 默认 | 描述 |
  |----------|------|------|
  | TERMINAL_SSH_PORT | 22 | SSH 端口 |
  | TERMINAL_SSH_KEY | (system default) | SSH 私钥 |
  | TERMINAL_SSH_PERSISTENT | true | 启用持久 shell |

  工作原理： 初始化时以 BatchMode=yes 和 StrictHostKeyChecking=accept-new 连接。持久 shell 会在远程主机上保持一个 bash -l 进程运行，并通过临时文件进行通信。需要 stdin_data 或 sudo 命令会自动回退到一次性模式。  

  
## 常见终端后端问题
  如果终端命令立即失败或终端工具被报告为已禁用：  
  - Local ——无特殊要求。入门时最安全的默认设置。
  - Docker — 运行 docker version 来验证 Docker 是否正常工作。如果失败，请修复 Docker 或 hermes config set terminal.backend local 。
  - SSH — 必须同时设置 TERMINAL_SSH_HOST 和 TERMINAL_SSH_USER 。如果缺少其中任何一个，Hermes 都会记录一个明确的错误日志。

## 拆卸时远程到主机的文件同步
  对于 SSH 、 Modal 和 Daytona 后端（代理的工作树位于与运行 Hermes 的主机不同的机器上的任何位置），Hermes 会跟踪代理在远程沙箱中修改的文件，并在会话拆除/沙箱清理时， 将修改后的文件同步回主机上的 ~/.hermes/cache/remote-syncs/<session-id>/ 。  

  - 触发条件：会话关闭、 /new 、 /reset 、网关消息超时、当子进程使用远程后端时 delegate_task 子代理完成。
  - 涵盖代理修改过的整个目录树，而不仅仅是它显式打开的文件。添加、编辑和删除操作都会被记录下来。
  - 当你查找时，远程沙箱可能已被拆除；本地 ~/.hermes/cache/remote-syncs/… 副本是代理更改内容的权威记录。
  - 大型二进制输出（模型检查点、原始数据集）的大小有限制——同步操作会跳过超过 file_sync_max_mb （默认值为 100 ）的文件。如果您预计会返回更大的文件，请增加此值。
    ```bash
    terminal:
      file_sync_max_mb: 100     # default — sync files up to 100 MB each
      file_sync_enabled: true   # default — set false to skip the sync entirely
    ```

  这样就可以从会话结束后会被销毁的临时云沙箱中恢复结果，而无需告诉代理显式地使用 scp 或 modal volume put 每个工件。  

## Docker 卷挂载
  使用 Docker 后端时， docker_volumes 允许您与容器共享主机目录。每个条目都使用标准的 Docker -v 语法： host_path:container_path[:options] 。  

  ```bash
  terminal:
    backend: docker
    docker_volumes:
      - "/home/user/projects:/workspace/projects"   # Read-write (default)
      - "/home/user/datasets:/data:ro"              # Read-only
      - "/home/user/.hermes/cache/documents:/output" # Gateway-visible exports
  ```

  这适用于：  
  - 向代理提供文件 （数据集、配置文件、参考代码）
  - 从代理接收文件 （生成的代码、报告、导出文件）
  - 共享工作区 ，您和代理都可以访问相同的文件

  如果您使用消息网关并希望代理通过以下方式发送生成的文件 MEDIA:/... ，建议使用专用的、主机可见的导出挂载点，例如： /home/user/.hermes/cache/documents:/output .  
  - 将 Docker 容器内的文件写入 /output/...
  - 在 MEDIA: 中输出主机路径 ，例如：  
    MEDIA:/home/user/.hermes/cache/documents/report.txt  
  - 除非主机上的网关进程也存在完全相同的路径，否则不要输出 /workspace/... 或 /output/... 路径。

  > [!WARNING]  
  > YAML 中重复的键会静默覆盖之前的键。如果您已经有一个 docker_volumes: 块，将新的挂载点合并到同一个列表中，而不是在文件中稍后添加另一个 docker_volumes: 键。  

  也可以通过环境变量进行设置： TERMINAL_DOCKER_VOLUMES='["/host:/container"]' （JSON 数组）。  

## Docker 凭证转发
  默认情况下，Docker 终端会话不会继承任意主机凭据。如果需要在容器内使用特定的令牌，请将其添加到 terminal.docker_forward_env 中。  
  ```bash
  terminal:
    backend: docker
    docker_forward_env:
      - "GITHUB_TOKEN"
      - "NPM_TOKEN"
  ```

  Hermes 首先解析当前 shell 中列出的每个变量，然后回退到 ~/.hermes/.env （如果它已保存，则使用 hermes config set 。  

  > [!WARNING]
  > docker_forward_env 中列出的所有内容都将对容器内运行的命令可见。请仅转发您认为可以安全地暴露给终端会话的凭据。

## 以主机用户身份运行容器
  默认情况下，Docker 容器以 root （UID 0）身份运行。在 /workspace 或其他绑定挂载点创建的文件最终会归主机上的 root 用户所有，因此会话结束后，您必须先使用 sudo chown 命令更改它们的所有者，才能从主机编辑器中编辑它们 terminal.docker_run_as_host_user 标志可以解决此问题： 
  ```bash
  terminal:
    backend: docker
    docker_run_as_host_user: true   # default: false
  ```

  启用 Hermes 后，它会在 docker run 命令后附加 --user $(id -u):$(id -g) ，这样写入绑定挂载目录（例如 /workspace 、 /root 以及 docker_volumes 中的任何目录）的文件将归主机用户所有，而不是 root 用户所有。缺点是：容器将无法再使用 apt install 或写入 root 用户拥有的路径（例如 /root/.npm ）。如果需要同时实现这两个功能，请使用 HOME 目录归非 root 用户所有的基础镜像（或者在镜像构建时添加所需的工具）。    

  为保持向后兼容，请将其保留为 false （默认值）。如果您的工作流程主要是“编辑已挂载的主机文件”，并且您厌倦了使用 sudo chown -R ，请启用此功能。  

## 可选：将启动目录挂载到 /workspace 目录
  Docker 沙箱默认保持隔离状态。除非您明确选择启用，否则 Hermes 不会将您当前主机的工作目录传递到容器中。  
  在 config.yaml 中启用它：  
  ```bash
  terminal:
    backend: docker
    docker_mount_cwd_to_workspace: true
  ```

  启用后：  
  - 如果您从 ~/projects/my-app 启动 Hermes，则该主机目录会绑定挂载到 /workspace
  - Docker 后端在 /workspace 目录下启动
  - 文件工具和终端命令都能看到同一个已挂载的项目

  禁用后， /workspace 仍由 sandbox 所有，除非您通过 docker_volumes 显式挂载某些内容。  
  安全性的权衡：  
  - false 保留沙箱边界
  - 启用 true 后，沙盒将直接访问您启动 Hermes 的目录。

  仅当您有意希望容器在实时主机文件上运行时才使用选择加入功能。  

## 持久 shell
  默认情况下，每个终端命令都在其自身的子进程中运行——工作目录、环境变量和 shell 变量会在命令之间重置。启用持久 shell 后，会保持一个长时间运行的 bash 进程在 execute() 调用之间持续运行，从而确保状态在命令之间保持不变。  

  这对于 SSH 后端尤其有用，因为它还能消除每个命令的连接开销。SSH 默认启用持久 shell，而本地后端则禁用持久 shell。  
  ```bash
  terminal:
    persistent_shell: true   # default — enables persistent shell for SSH
  ```

  禁用方法：  
  ```bash
  hermes config set terminal.persistent_shell false
  ```

  命令之间保持不变的内容：  
  - 工作目录（ cd /tmp 保留用于下一个命令）
  - 导出环境变量（ export FOO=bar ）
  - Shell 变量（ MY_VAR=hello ）

  优先权：  
  | Level | Variable | Default |
  |-------|----------|---------|
  | Config | terminal.persistent_shell | true |
  | SSH override | TERMINAL_SSH_PERSISTENT | 遵循配置 |
  | Local override | TERMINAL_LOCAL_PERSISTENT | false |

  每个后端环境变量的优先级最高。如果您也希望在本地后端拥有持久 shell：  
  ```bash
  export TERMINAL_LOCAL_PERSISTENT=true
  ```

  > [!NOTE]  
  > 需要 stdin_data 或 sudo 的命令会自动回退到一次性模式，因为持久 shell 的 stdin 已经被 IPC 协议占用。  

# 技能设置
  技能可以通过其 SKILL.md 前置元数据声明自己的配置设置。这些是存储在 config.yaml 中 skills.config 命名空间下的非秘密值（路径、首选项、域设置）。  

  ```bash
  skills:
    config:
      myplugin:
        path: ~/myplugin-data   # Example — each skill defines its own keys
  ```

  技能设置如何运作：  
  - hermes config migrate 会扫描所有已启用的技能，查找未配置的设置，并提示您进行配置。
  - hermes config show 会在“技能设置”下显示所有技能设置及其所属技能。
  - 技能加载时，其解析后的配置值会自动注入到技能上下文中。

  手动设置值：  
  ```bash
  hermes config set skills.config.myplugin.path ~/myplugin-data
  ```

## 对代理创建的技能写入​​进行保护
  当代理使用 skill_manage 创建、编辑、修补或删除技能时，Hermes 可以选择扫描新增/更新的内容，以查找危险的关键字模式（例如凭证窃取、明显的提示注入、数据泄露指令）。扫描器默认处于关闭状态 ——实际代理工作流程中，如果合法地访问 ~/.ssh/ 或提及 $OPENAI_API_KEY ，则触发启发式规则的频率过高。如果您希望在代理的技能写入​​数据之前收到扫描器的提示，请重新启用此功能：  

  ```bash
  skills:
    guard_agent_created: true   # default: false
  ```

  启用此功能后，任何被标记为 skill_manage 写入操作都会显示为审批提示，并附有扫描器给出的理由。已接受的写入操作会成功执行；被拒绝的写入操作会向代理返回一个解释性错误。  

# 记忆配置
  | 特性 | MEMORY.md | USER.md |
  |------|-----------|---------|
  | 定位与角色 | Agent 的工作笔记 (Agent's Notes)：记录 Agent 为更好地完成任务所需要的环境和项目信息。 | 用户画像 (User Profile)：存储关于用户是谁、偏好和习惯的信息，用以辅助 Agent 理解用户。 |
  | 记录内容 | 客观、元数据：项目配置、服务器信息、工具特性、工作流经验等。 | 主观、个性化：用户姓名、沟通偏好、技术能力、雷区、输出格式要求等。 |
  | 容量限制 | 约 2,200 字符 (≈ 800 tokens)，同时也有另一种说法是严格限制在 800 tokens。 | 约 1,375 字符 (≈ 500 tokens)。 |
  | 核心目的 | 记住“事情该怎么做”，理解工作上下文，提升任务执行效率。 | 记住“我是谁”，调整交互风格，提供更个性化的服务。 |
  
  ```bash
  memory:
    memory_enabled: true
    user_profile_enabled: true
    memory_char_limit: 2200   # ~800 tokens
    user_char_limit: 1375     # ~500 tokens
  ```

## 协同工作机制
  - 存储路径：都位于 ~/.hermes/memories/ 目录下。
  - 生效机制：它们都在每次新会话开始时被加载，并作为“冻结快照 (frozen snapshot)”注入到系统提示中。
  - 时效性：在会话中对文件所做的任何修改（添加/删除），都会立即写入磁盘，但只会在下一次新会话中生效。这种设计是为了保持 LLM 前缀缓存的稳定性。

   Agent 自身 在对话中动态地管理这些记忆文件，其操作类型包括 add（添加）、replace（替换）、remove（删除）。  
   此外，用户也可以在任何时候使用文本编辑器手动修改这两个文件。  

# 文件读取安全
  控制单次 read_file 调用可以返回的内容量。超出限制的读取操作将被拒绝，并返回错误信息，告知代理使用更小的 offset 和 limit 。这可以防止对压缩后的 JS 包或大型数据文件的单次读取操作导致上下文窗口被大量数据淹没。  

  ```bash
  file_read_max_chars: 100000  # default — ~25-35K tokens
  ```

  如果模型上下文窗口较大且经常读取大文件，则应提高此值。对于上下文窗口较小的模型，则应降低此值以保持读取效率：  
  ```bash
  # Large context model (200K+)
  file_read_max_chars: 200000
  
  # Small local model (16K context)
  file_read_max_chars: 30000
  ```

  该代理还会自动进行文件去重——如果同一文件区域被读取两次且文件内容未发生更改，则会返回一个轻量级存根，而不是重新发送文件内容。此机制会在上下文压缩后重置，以便代理能够在文件内容被压缩后重新读取文件。  

# 上下文压缩
  Hermes 会自动压缩过长的对话，使其保持在模型上下文窗口内。压缩摘要器是一个独立的 LLM 调用——您可以将其指向任何提供商或端点。  

  所有压缩设置都位于 config.yaml 中（没有环境变量）。  

## 完整参考
  ```bash
  compression:
    enabled: true                                     # Toggle compression on/off
    threshold: 0.50                                   # Compress at this % of context limit
    target_ratio: 0.20                                # Fraction of threshold to preserve as recent tail
    protect_last_n: 20                                # Min recent messages to keep uncompressed
    hygiene_hard_message_limit: 400                   # Gateway safety valve — see below
  
  # 摘要模型/提供程序在辅助配置下进行配置：
  auxiliary:
    compression:
      model: "google/gemini-3-flash-preview"          # Model for summarization
      provider: "auto"                                # Provider: "auto", "openrouter", "nous", "codex", "main", etc.
      base_url: null                                  # Custom OpenAI-compatible endpoint (overrides provider)
  ```

  > [!WARNING]
  > 传统配置迁移  
  > 首次加载时（配置版本 17），包含 compression.summary_model 、 compression.summary_provider 和 compression.summary_base_url 的旧配置会自动迁移到 auxiliary.compression.* 无需手动操作。  

  hygiene_hard_message_limit 是一个仅针对网关的预压缩安全阀 。包含数千条消息的失控会话可能会在正常的上下文百分比阈值触发之前就达到模型上下文限制；当消息数量超过此上限时，Hermes 会强制压缩，而不管令牌的使用情况如何。默认值为 400 对于会话时间很长的平台，可以提高此值；降低此值可以强制执行更严格的压缩。在运行中的网关上编辑此值，将在下一条消息中生效（见下文）。  

  > [!IMPORTANT]  
  > 压缩和上下文长度的网关热重载  
  > 从最新版本开始，在运行中的网关上编辑 config.yaml 文件中的 model.context_length 或任何 compression.* 键，会在下一条消息时生效——无需重启网关、无需 /reset 、无需会话轮换。缓存代理的签名包含这些键，因此网关会在检测到更改时透明地重建代理。API 密钥和工具/技能配置仍然需要使用常规的重新加载路径。  

## 常见配置
  默认（自动检测）——无需配置：  
  ```bash
  compression:
    enabled: true
    threshold: 0.50
  ```

  使用您的主要提供商和主要型号。如果您想使用比主聊天型号更便宜的型号进行压缩，请针对每个任务进行覆盖（例如， auxiliary.compression.provider: openrouter + model: google/gemini-2.5-flash ）。  

  强制使用特定提供商 （基于 OAuth 或 API 密钥）：  
  ```bash
  auxiliary:
    compression:
      provider: nous
      model: gemini-3-flash
  ```
  - 可与任何网络服务提供商配合使用： nous 、 openrouter 、 codex 、 anthropic 、 main 等。

  自定义端点 （自托管、Ollama、Zai、DeepSeek 等）：  
  ```bash
  auxiliary:
    compression:
      model: glm-4.7
      base_url: https://api.z.ai/api/coding/paas/v4
  ```

  指向自定义的 OpenAI 兼容端点。使用 OPENAI_API_KEY 进行身份验证。  

## 三个旋钮如何相互作用
  | auxiliary.compression.provider | auxiliary.compression.base_url | Result |
  |-------------------------------|-------------------------------|--------|
  | auto (default) | not set | 自动检测最佳可用提供商 |
  | nous / openrouter / etc. | not set | 强制使用该提供商，并使用其身份验证 |
  | any | set | 直接使用自定义端点（忽略提供程序） |

  > [!WARNING]  
  > 摘要模型上下文长度要求  
  > 摘要模型的上下文窗口必须至少与主代理模型的上下文窗口一样大。压缩器会将对话的中间部分完整地发送给摘要模型——如果该模型的上下文窗口小于主模型的上下文窗口，则摘要调用将因上下文长度错误而失败。发生这种情况时，中间部分的内容将被丢弃，没有摘要 ，对话上下文将悄无声息地丢失。如果您重写了该模型，请验证其上下文长度是否等于或大于主模型的上下文长度。  

# 上下文引擎
  上下文引擎控制着当模型接近词元限制时如何管理对话。内置的 compressor​​引擎使用有损摘要（参见上下文压缩 ）。插件引擎可以用其他策略替换它。  

  ```bash
  context:
    engine: "compressor"    # default — built-in lossy summarization
  ```

  要使用插件引擎（例如，用于无损上下文管理的 LCM）：  
  ```bash
  context:
    engine: "lcm"          # must match the plugin's name
  ```

  插件引擎永远不会自动激活 ——您必须显式地将 context.engine 设置为插件名称。您可以通过 hermes plugins → 提供商插件 → 上下文引擎 浏览和选择可用的引擎。  

# 凭证池策略
  当您拥有同一提供商的多个 API 密钥或 OAuth 令牌时，请配置轮换策略：  
  ```bash
  credential_pool_strategies:
    openrouter: round_robin    # cycle through keys evenly
    anthropic: least_used      # always pick the least-used key
  ```
  - 选项： fill_first （默认）、 round_robin 、 least_used 、 random 。

## 静态消息的 UI 语言
  display.language 设置会翻译一小部分静态的面向用户的消息，例如 CLI 批准提示、一些网关斜杠命令回复（例如 restart-drain 通知、“批准已过期”、“目标已清除”）。它不会翻译代理响应、日志行、工具输出、错误回溯或斜杠命令描述——这些内容仍保持英文。如果您希望代理本身以其他语言回复，只需在提示或系统消息中指定即可。

  支持的值： en （默认）、 zh （简体中文）、 ja （日语）、 de （德语）、 es （西班牙语）、 fr （法语）、 tr （土耳其语）、 uk （乌克兰语）。未知值将回退到英语。  

  您还可以使用 HERMES_LANGUAGE 环境变量为每个会话设置此值，该变量会覆盖配置值。  
  ```bash
  display:
    language: zh   # CLI approval prompts appear in Chinese
  ```

  | Mode | What you see |
  |------|--------------|
  | off | 沉默——这才是最终的回应 |
  | new | 仅当工具更换时才会显示工具指示器 |
  | all | 每次工具调用都带有简短预览（默认） |
  | verbose | 完整参数、结果和调试日志 |

  在命令行界面 (CLI) 中，可以使用 /verbose 命令切换这些模式。要在消息平台（例如 Telegram、Discord、Slack 等）中使用 /verbose 命令，请在上面的 display 设置中将 tool_progress_command: true 。该命令随后会切换模式并保存到配置中。  

## 运行时元数据页脚（仅限网关）
  当 display.runtime_metadata_footer: true 为真时，Hermes 会在每个网关回合的最后一条消息后添加一个简短的运行时上下文页脚——其中包含与 CLI 状态栏中显示的信息相同的信息（模型、会话持续时间、令牌、成本）。默认情况下此功能处于关闭状态；如果您的团队希望每个回复都包含来源信息，则可以为每个网关单独启用此功能。  

  ```bash
  display:
    runtime_metadata_footer: true
  ```

  添加到 Telegram/Discord/Slack 回复中的示例页脚：  
  ```bash
  — claude-opus-4.7 · 12 tool calls · 2m 14s · $0.042
  ```

  只有回合的最后一条消息才会显示在页脚；中间的更新保持空白。  

## 每个平台的进度覆盖
  不同的平台对消息详细程度的需求不同。例如，Signal 无法编辑消息，因此每次进度更新都会生成一条单独的消息——这会造成信息过载。可以使用 display.platforms 设置每个平台的显示模式：  
  ```bash
  display:
    tool_progress: all          # global default
    platforms:
      signal:
        tool_progress: 'off'    # silence progress on Signal
      telegram:
        tool_progress: verbose  # detailed progress on Telegram
      slack:
        tool_progress: 'off'    # quiet in shared Slack workspace
  ```

  没有覆盖设置的平台将回退到全局的 tool_progress 值。有效的平台键包括： telegram 、 discord 、 slack 、 signal 、 whatsapp 、 matrix 、 mattermost 、 email 、 sms 、 homeassistant 、 dingtalk 、 feishu 、 wecom 、 weixin 、 bluebubbles 和 qqbot 。旧版的 display.tool_progress_overrides 键仍然会加载以保持向后兼容性，但已被弃用，并在首次加载时迁移到 display.platforms 中。  

  interim_assistant_messages 仅限网关使用。启用后，Hermes 会将已完成的转弯助手更新作为单独的聊天消息发送。这与 tool_progress 无关，也不需要网关流传输。

# 语音转文本 (STT)
  ```bash
  stt:
    provider: "local"            # "local" | "groq" | "openai" | "mistral"
    local:
      model: "base"              # tiny, base, small, medium, large-v3
    openai:
      model: "whisper-1"         # whisper-1 | gpt-4o-mini-transcribe | gpt-4o-transcribe
    # model: "whisper-1"         # Legacy fallback key still respected
  ```

  提供商行为：  
  - local 使用运行在您机器上的 faster-whisper 。请使用 pip install faster-whisper 单独安装它。
  - groq 使用 Groq 的 Whisper 兼容端点并读取 GROQ_API_KEY 。
  - openai 使用 OpenAI 语音 API 并读取 VOICE_TOOLS_OPENAI_KEY

  如果请求的提供商不可用，Hermes 将按以下顺序自动回退： local → groq → openai 。  

  Groq 和 OpenAI 模型覆盖是环境驱动的：  
  ```bash
  STT_GROQ_MODEL=whisper-large-v3-turbo
  STT_OPENAI_MODEL=whisper-1
  GROQ_BASE_URL=https://api.groq.com/openai/v1
  STT_OPENAI_BASE_URL=https://api.openai.com/v1
  ```
