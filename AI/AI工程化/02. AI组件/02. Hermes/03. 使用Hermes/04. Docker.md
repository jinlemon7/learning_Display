Docker 与 Hermes Agent 的交互方式主要有两种：  
1. 在 Docker 中运行 Hermes —— 代理本身在容器内运行（本页主要内容）
2. Docker 作为终端后端 —— 代理程序在您的主机上运行，​​但每个命令都在单个持久的 Docker 沙箱容器内执行，该容器在 Hermes 进程的生命周期内，在工具调用、 /new 和子代理程序之间持续存在

本页介绍方案一。容器将所有用户数据（配置、API 密钥、记忆、会话、技能）存储在Docker主机挂载的单个目录 /opt/data 中。镜像本身是无状态的，可以通过拉取新版本进行升级，而不会丢失任何配置。

# 配置Docker Compose
## 方案一：使用 Docker Compose
### 创建项目构建目录：
  ```bash
  # 不建议自己创建数据目录，`docker compose up -d` 后 Hermes 会自己创建该目录
  # mkdir -p ~/.hermes

  mkdir -p ~/hermes-agent
  cd ~/hermes-agent
  ```

### 配置 docker-compose.yml 文件：
  在 ~/hermes-agent 目录下，修改名为 docker-compose.yml 的文件，并将以下内容完整复制进去：  
  ```bash
  services:
    # Hermes Agent 网关服务
    gateway:
      image: nousresearch/hermes-agent:latest
      container_name: hermes-gateway
      restart: unless-stopped
      command: ["gateway", "run"]
      ports:
        - "8642:8642" # 映射Agent API端口 
      volumes:
        - ~/.hermes:/opt/data # 挂载数据目录，确保持久化存储
      networks:
        - hermes-net
      deploy:
        resources:
          limits:
            memory: 4G
            cpus: "1.0"

    # Hermes Dashboard 控制面板服务
    dashboard:
      image: nousresearch/hermes-agent:latest
      container_name: hermes-dashboard
      restart: unless-stopped
      command: ["dashboard", "--host", "0.0.0.0", "--insecure"]
      ports:
        - "9119:9119"   # 映射Dashboard访问端口
      volumes:
        - ~/.hermes:/opt/data # 共享数据目录
      networks:
        - hermes-net
      deploy:
        resources:
          limits:
            memory: 512M
            cpus: "0.25"
      depends_on:
        - gateway # 确保Agent网关服务先启动 
      environment:
        - GATEWAY_HEALTH_URL=http://gateway:8642 # 指向Agent网关的健康检查地址
    
  networks:
    hermes-net:
      driver: bridge
  ```
  - 安全提示：如果服务器有公网IP，请务必为 Dashboard 设置访问密码。只需在启动命令中添加环境变量即可：- DASHBOARD_PASSWORD=你的强密码。

### 启动服务：
  在 docker-compose.yml 文件所在目录下，运行以下命令：  
  ```bash
  docker compose up -d
  ```

### 修改 docker-compose.yml 后重启服务
  ```bash
  docker compose down && docker compose up -d
  ```

### 容器内部配置
  进入容器终端：  
  ```bash
  docker exec -it hermes-gateway bash
  ```

  创建软链接,让容器重启后也能直接识别 hermes 命令(新容器只需执行1次)。  
  ```bash
  ln -s /opt/hermes/.venv/bin/hermes /usr/local/bin/hermes
  ```

  配置Hermes：  
  ```bash
  hermes setup
  ```

  内部配置完，重启gateway
  ```bash
  # 退出容器终端
  exit
  
  docker restart hermes-gateway
  ```

  此时直接执行 docker exec -it hermes bash 会立刻闪退，回到宿主机：  
  更改 ~/.hermes 的角色  
  
  在宿主机执行：  
  ```bash
  sudo chown -R 10000:10000 ~/.hermes
  ```

### 把 Docker socket 挂载进容器，让 Hermes 完全控制宿主机
  找到对应的 docker-compose.yml，在 volumes: 下面添加一行：  
  ```bash
  volumes:
    - /var/run/docker.sock:/var/run/docker.sock
  ```

  开放 socket 权限:  
  ```bash
  sudo chmod 666 /var/run/docker.sock
  ```

  然后重启:    
  ```bash
  docker compose down && docker compose up -d
  ```

### 访问控制面板：
  配置云服务器的防火墙规则：8642、9119端口。  
  服务启动后，在浏览器中访问 http://你的服务器IP:9119 即可进入控制面板。

# 快速入门
  如果这是您第一次运行 Hermes Agent，请在主机上创建一个数据目录，并以交互方式启动容器以运行设置向导：  
  ```bash
  mkdir -p ~/.hermes
  docker run -it --rm \
    -v ~/.hermes:/opt/data \
    nousresearch/hermes-agent setup
  ```

  这将引导您进入设置向导，该向导会提示您输入 API 密钥并将其写入 ~/.hermes/.env 文件。您只需执行此操作一次。强烈建议您此时为网关设置一个聊天系统。  

# 以网关模式运行
  配置完成后，将容器作为持久网关（Telegram、Discord、Slack、WhatsApp 等）在后台运行：  
  ```bash
  docker run -d \
    --name hermes \
    --restart unless-stopped \
    -v ~/.hermes:/opt/data \
    -p 8642:8642 \
    nousresearch/hermes-agent gateway run
  ```

  端口 8642 暴露了网关的 OpenAI 兼容 API 服务器和健康检查端点。如果您仅使用聊天平台（例如 Telegram、Discord 等），则此端口为可选；但如果您希望仪表盘或外部工具能够访问网关，则此端口为必需。  

  注意：API 服务器的启用取决于 API_SERVER_ENABLED=true 。要将其暴露给容器内 127.0.0.1 以外的地址，还需要设置 API_SERVER_HOST=0.0.0.0 和一个 API_SERVER_KEY （至少 8 个字符——可以使用 openssl rand -hex 32 生成）。示例：  
  ```bash
  docker run -d \
    --name hermes \
    --restart unless-stopped \
    -v ~/.hermes:/opt/data \
    -p 8642:8642 \
    -e API_SERVER_ENABLED=true \
    -e API_SERVER_HOST=0.0.0.0 \
    -e API_SERVER_KEY=your_api_key_here \
    -e API_SERVER_CORS_ORIGINS='*' \
    nousresearch/hermes-agent gateway run
  ```
  - 在面向互联网的计算机上打开任何端口都存在安全风险。除非您了解这些风险，否则不应这样做。

# 运行仪表板
  内置的 Web 控制面板作为可选的辅助进程运行在与网关相同的容器内。设置 HERMES_DASHBOARD=1 并将端口 9119 与网关的端口 8642 一起暴露出来：  
  ```bash
  docker run -d \
    --name hermes \
    --restart unless-stopped \
    -v ~/.hermes:/opt/data \
    -p 8642:8642 \
    -p 9119:9119 \
    -e HERMES_DASHBOARD=1 \
    nousresearch/hermes-agent gateway run
  ```

  入口点会在 exec 主命令之前，在后台启动 hermes dashboard （以非 root hermes 身份运行）。控制面板的输出在 docker logs 中会以 [dashboard] 为前缀，以便与网关日志轻松区分。  
  | Environment variable | 描述 | 默认 |
  |----------------------|------|------|
  | HERMES_DASHBOARD | 设置为 1 （或 true / yes ）可在主命令旁边启动仪表板 | (未设置——仪表盘未启动) |
  | HERMES_DASHBOARD_HOST | 仪表盘 HTTP 服务器的绑定地址 | 0.0.0.0 |
  | HERMES_DASHBOARD_PORT | 仪表盘 HTTP 服务器端口 | 9119 |
  | HERMES_DASHBOARD_TUI | 设置为 1 以显示浏览器内的聊天选项卡（通过 PTY/WebSocket 嵌入 hermes --tui ） | (未设置) |

  默认情况下，主机必须设置为 HERMES_DASHBOARD_HOST=0.0.0.0 ，才能通过发布的端口访问仪表板；在这种情况下，入口点会自动向 hermes dashboard 传递 --insecure 。如果您希望将仪表板的访问限制在容器内（例如，在 Sidecar 容器的反向代理之后），请将其覆盖为 127.0.0.1 。  

  > [!NOTE]  
  > 仪表盘侧进程不受监控 ——如果它崩溃，则会一直处于关闭状态，直到容器重启。不支持将其作为单独的容器运行：仪表盘的网关存活检测需要与网关进程共享 PID 命名空间。

# 持续性卷
  /opt/data 卷是 Hermes 所有状态的唯一数据源。它映射到主机上的 ~/.hermes/ 目录，内容包括：  
  ```bash
  | Path | Contents |
  |------|----------|
  | .env | API keys and secrets / API 密钥和密钥 |
  | config.yaml | All Hermes configuration / 所有Hermes配置 |
  | SOUL.md | Agent personality/identity / Agent个性/身份 |
  | sessions/ | Conversation history / 对话历史 |
  | memories/ | Persistent memory store / 持久记忆存储 |
  | skills/ | Installed skills / 已安装技能 |
  | cron/ | Scheduled job definitions / 定时任务定义 |
  | hooks/ | Event hooks / 事件钩子 |
  | logs/ | Runtime logs / 运行时日志 |
  | skins/ | Custom CLI skins / 自定义 CLI 皮肤 |
  ```

  > [!WARNING]  
  > 切勿同时对同一数据目录运行两个 Hermes 网关容器——会话文件和内存存储并非为并发写入访问而设计。  

# 多配置文件支持
  Hermes 支持多配置文件 —— 即独立的 ~/.hermes/ 目录，允许您从单个Hermes配置文件安装运行独立的代理（不同的 凭据、SOUL、记忆、会话和技能）。 在 Docker 环境下运行时，不建议使用 Hermes 内置的多配置文件功能。  

  相反，推荐的做法是每个配置文件对应一个容器 ，每个容器将其自身的主机目录绑定挂载为 /opt/data ：  
  ```bash
  # Work profile
  docker run -d \
    --name hermes-work \
    --restart unless-stopped \
    -v ~/.hermes-work:/opt/data \
    -p 8642:8642 \
    nousresearch/hermes-agent gateway run
  
  # Personal profile
  docker run -d \
    --name hermes-personal \
    --restart unless-stopped \
    -v ~/.hermes-personal:/opt/data \
    -p 8643:8642 \
    nousresearch/hermes-agent gateway run
  ```

  为什么 Docker 中要使用不同的配置文件来区分容器：  
  - 隔离性 —— 每个容器都有自己的文件系统、进程表和资源限制。一个容器中的崩溃、依赖项更改或失控会话不会影响其他容器。
  - 独立生命周期 —— 分别升级、重启、暂停或回滚每个代理（ docker restart hermes-work 不会影响 hermes-personal ）。
  - 端口和网络完全隔离 —— 每个网关都绑定自己的主机端口；聊天平台或 API 服务器之间不存在串扰风险。
  - 更简单的思维模型 —— 容器即配置文件。备份、迁移和权限都遵循绑定挂载的目录，无需记住额外的 --profile 标志。
  - 避免并发写入风险 —— 上面关于永远不要对同一数据目录运行两个网关的警告仍然适用于单个容器内的配置文件。

  在 Docker Compose 中，这意味着为每个配置文件声明一个服务，并为其指定不同的 container_name 、 volumes 和 ports ：  
  ```bash
  services:
    hermes-work:
      image: nousresearch/hermes-agent:latest
      container_name: hermes-work
      restart: unless-stopped
      command: gateway run
      ports:
        - "8642:8642"
      volumes:
        - ~/.hermes-work:/opt/data
  
    hermes-personal:
      image: nousresearch/hermes-agent:latest
      container_name: hermes-personal
      restart: unless-stopped
      command: gateway run
      ports:
        - "8643:8642"
      volumes:
        - ~/.hermes-personal:/opt/data
  ```

# 环境变量转发
  API 密钥是从容器内的 /opt/data/.env 文件中读取的。您也可以直接传递环境变量：  
  ```bash
  docker run -it --rm \
    -v ~/.hermes:/opt/data \
    -e ANTHROPIC_API_KEY="sk-ant-..." \
    -e OPENAI_API_KEY="sk-..." \
    nousresearch/hermes-agent
  ```

  直接使用 -e 标志会覆盖 .env 中的值。这对于 CI/CD 或 secrets-manager 集成非常有用，因为在这些情况下，您不希望密钥保存在磁盘上。  

# Docker Compose 示例
  对于需要同时部署网关和控制面板的持久化部署， docker-compose.yaml 文件会很方便：  

  ```bash
  services:
    hermes:
      image: nousresearch/hermes-agent:latest
      container_name: hermes
      restart: unless-stopped
      command: gateway run
      ports:
        - "8642:8642"   # gateway API
        - "9119:9119"   # dashboard (only reached when HERMES_DASHBOARD=1)
      volumes:
        - ~/.hermes:/opt/data
      environment:
        - HERMES_DASHBOARD=1
        # Uncomment to forward specific env vars instead of using .env file:
        # - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
        # - OPENAI_API_KEY=${OPENAI_API_KEY}
        # - TELEGRAM_BOT_TOKEN=${TELEGRAM_BOT_TOKEN}
      deploy:
        resources:
          limits:
            memory: 4G
            cpus: "2.0"
  ```

  首先运行 docker compose up -d ，然后运行 docker compose logs -f 查看日志。仪表盘输出以 [dashboard] 为前缀，因此可以轻松地从网关日志中进行筛选。  

# 资源限制
  Hermes 容器需要一定的资源。建议最低配置：  
  | Resource | Minimum | Recommended |
  |----------|---------|-------------|
  | Memory | 1 GB | 2–4 GB |
  | CPU | 1 个核心 | 2 个核心 |
  | Disk (data volume) | 500 MB | 2+ GB（随会话/技能提升而增长） |


  浏览器自动化（Playwright/Chromium）是最占用内存的功能。如果您不需要浏览器工具，1 GB 内存就足够了。如果启用浏览器工具，则至少需要 2 GB 内存。  

  设置 Docker 限制：  
  ```bash
  docker run -d \
    --name hermes \
    --restart unless-stopped \
    --memory=4g --cpus=2 \
    -v ~/.hermes:/opt/data \
    nousresearch/hermes-agent gateway run
  ```

# Dockerfile 的作用
  官方镜像基于 debian:13.4 ，包含以下内容  
  - Python 3 及其所有 Hermes 依赖项（ uv pip install -e ".[all]" ）
  - Node.js + npm（用于浏览器自动化和 WhatsApp 桥接）
  - 使用 Chromium 安装 Playwright（ npx playwright install --with-deps chromium --only-shell ）
  - ripgrep、ffmpeg、git 和 tini 作为系统实用程序
  - docker-cli — 因此，在容器内运行的代理可以驱动主机的 Docker 守护程序（通过绑定挂载 /var/run/docker.sock 来选择加入），以执行 docker build 、 docker run 、容器检查等操作。
  - openssh-client — 启用容器内部的 SSH 终端后端 。SSH 后端会调用系统 ssh 二进制文件；如果没有它，容器化安装时会静默失败。
  - WhatsApp 桥接器（ scripts/whatsapp-bridge/ ）

  入口脚本（ docker/entrypoint.sh ）在首次运行时引导数据卷：  
  - 创建目录结构（ sessions/ 、 memories/ 、 skills/ 等）
  - 如果 .env.example .env 存在，则将其复制到 .env
  - 如果默认的 SOUL.md 缺失，则复制该文件。
  - 使用基于清单的方法同步捆绑技能（保留用户编辑）
  - 当 HERMES_DASHBOARD=1 时，可以选择将 hermes dashboard 作为后台进程启动
  - 然后使用你传递的任何参数运行 hermes

  > [!WARNING]
  > 除非您将 /opt/hermes/docker/entrypoint.sh docker/entrypoint.sh` 保留在命令链中，否则请勿覆盖镜像入口点。入口点会在创建网关状态文件之前将 root 权限降至 hermes 用户。默认情况下，官方镜像会拒绝以 root 用户身份 hermes gateway run ，因为这可能会导致 /opt/data 目录下留下 root 用户拥有的文件，从而破坏后续的仪表盘或网关启动。仅当您有意承担此风险时，才应设置 HERMES_ALLOW_ROOT_GATEWAY=1 。

# 技能和证书文件
  当使用 Docker 作为执行环境时（非本章主要介绍方案，而是代理在 Docker 沙箱中运行命令——参见 “配置”→“Docker 后端 ”），Hermes 会为所有工具调用重用一个长期存在的容器，并自动将技能目录（ ~/.hermes/skills/ ）以及技能声明的任何凭据文件作为只读卷挂载到该容器中。技能脚本、模板和引用无需手动配置即可在沙箱中使用。由于容器在 Hermes 进程的生命周期内持续存在，因此您安装的任何依赖项或编写的任何文件都会保留到下次工具调用时。  

  SSH 和 Modal 后端也进行相同的同步 —— 技能和凭证文件在每次命令执行之前通过 rsync 或 Modal 挂载 API 上传。  

# 连接到本地推理服务器（vLLM、Ollama 等）
  当在 Docker 中运行 Hermes，并且您的推理服务器（vLLM、Ollama、text-generation-inference 等）也在主机上或另一个容器中运行时，网络需要格外注意。  

## Docker Compose（推荐）
  将这两个服务放在同一个 Docker 网络中。这是最可靠的方法：  
  ```bash
  services:
    vllm:
      image: vllm/vllm-openai:latest
      container_name: vllm
      command: >
        --model Qwen/Qwen2.5-7B-Instruct
        --served-model-name my-model
        --host 0.0.0.0
        --port 8000
      ports:
        - "8000:8000"
      networks:
        - hermes-net
      deploy:
        resources:
          reservations:
            devices:
              - capabilities: [gpu]
  
    hermes:
      image: nousresearch/hermes-agent:latest
      container_name: hermes
      restart: unless-stopped
      command: gateway run
      ports:
        - "8642:8642"
      volumes:
        - ~/.hermes:/opt/data
      networks:
        - hermes-net
  
  networks:
    hermes-net:
      driver: bridge
  ```

  然后，在你的 ~/.hermes/config.yaml 中，使用容器名称作为主机名：  
  ```bash
  model:
    provider: custom
    model: my-model
    base_url: http://vllm:8000/v1
    api_key: "none"
  ```

  > [!IMPORTANT]  
  > 要点  
  > 使用容器名称 （ vllm ）作为主机名，而不是 localhost 或 127.0.0.1 ，因为它们指的是 Hermes 容器本身。  
  > model 值必须与您传递给 vLLM 的 --served-model-name 匹配。  
  > 将 api_key 设置为任何非空字符串（vLLM 需要标头，但默认情况下不对其进行验证）。  
  > base_url 中不要包含尾部斜杠。  

## 独立运行 Docker（无需 Compose）
  如果您的推理服务器直接运行在主机上（而不是在 Docker 中），请在 macOS/Windows 上使用 host.docker.internal ，或在 Linux 上使用 --network host ：  

### macOS / Windows：  
  ```bash
  docker run -d \
    --name hermes \
    -v ~/.hermes:/opt/data \
    -p 8642:8642 \
    nousresearch/hermes-agent gateway run
  ```

  ```bash
  # config.yaml
  model:
    provider: custom
    model: my-model
    base_url: http://host.docker.internal:8000/v1
    api_key: "none"
  ```

 ### Linux（主机网络）：
   ```bash
  docker run -d \
    --name hermes \
    --network host \
    -v ~/.hermes:/opt/data \
    nousresearch/hermes-agent gateway run
  ```

  ```bash
  # config.yaml
  model:
    provider: custom
    model: my-model
    base_url: http://127.0.0.1:8000/v1
    api_key: "none"
  ```

  > [!WARNING]  
  > 使用 --network host 时， -p 标志将被忽略——所有容器端口都将直接暴露在主机上。  

## 验证连接性
  在 Hermes 容器内部，确认推理服务器可访问：  
  ```bash
  docker exec hermes curl -s http://vllm:8000/v1/models
  ```
  您应该会看到一个列出您所提供服务的模型的 JSON 响应。如果此操作失败，请检查：  
  1. 两个容器位于同一个 Docker 网络中（ docker network inspect hermes-net ）。
  2. 推理服务器正在监听 0.0.0.0 ，而不是 127.0.0.1
  3. 端口号匹配

## Ollama
  Ollama 的工作方式与其他本地推理服务器相同。如果 Ollama 运行在主机上，请使用 host.docker.internal:11434 （macOS/Windows）或 127.0.0.1:11434 ，需使用 --network host ）。如果 Ollama 运行在同一 Docker 网络上的独立容器中：  

  ```bash
  model:
    provider: custom
    model: llama3
    base_url: http://ollama:11434/v1
    api_key: "none"
  ```

# 故障排除

## Docker 容器立即退出
  检查日志： docker logs hermes 。常见原因：  
  - 缺少或无效的 .env 文件 — 请先以交互方式运行以完成设置。
  - 如果运行过程中暴露了端口，则会出现端口冲突。

## "Permission denied" 错误
  容器入口点通过 gosu 将权限降低到非 root 用户 hermes （UID 10000）。如果您的主机 ~/.hermes/ 目录的所有者是其他 UID，请将 HERMES_UID / HERMES_GID 设置为与您的主机用户匹配，或者确保数据目录可写：  
  ```bash
  chmod -R 755 ~/.hermes
  ```

## 浏览器工具无法正常工作
  Playwright 需要共享内存。请在 Docker 运行命令中添加 --shm-size=1g ：  
  ```bash
  docker run -d \
    --name hermes \
    --shm-size=1g \
    -v ~/.hermes:/opt/data \
    nousresearch/hermes-agent gateway run
  ```

## 网络问题后网关无法重新连接
  --restart unless-stopped 标志可以处理大多数瞬态故障。如果网关卡住，请重启容器：  
  ```bash
  docker restart hermes
  ```

## 检查容器健康状况
  ```bash
  docker logs --tail 50 hermes          # Recent logs
  docker run -it --rm nousresearch/hermes-agent:latest version     # Verify version
  docker stats hermes                    # Resource usage
  ```
