Hermes Agent 会自动将每次对话保存为一个会话。会话支持对话恢复、跨会话搜索和完整的对话历史记录管理。  

# 会话如何运作
  无论对话来自命令行界面 (CLI)、Telegram、Discord、Slack、WhatsApp、Signal、Matrix、Teams 还是任何其他消息平台，所有对话都会以会话的形式存储，并包含完整的消息历史记录。会话在两个互补的系统中进行跟踪：  
  - SQLite 数据库 （ ~/.hermes/state.db ）——结构化会话元数据，支持 FTS5 全文搜索
  - JSONL 转录 （ ~/.hermes/sessions/ ）— 包含工具调用（网关）的原始对话转录

  SQLite 数据库存储：  
  - 会话 ID、源平台、用户 ID
  - 会议标题 （唯一且易于理解的名称）
  - 模型名称和配置
  - 系统提示符快照
  - 完整消息历史记录（角色、内容、工具调用、工具结果）
  - 令牌计数（输入/输出）
  - 时间戳（开始时间，结束时间）
  - 父会话 ID（用于压缩触发的会话拆分）

# CLI 会话恢复
  使用 --continue 或 --resume 从 CLI 恢复之前的对话：  

## 继续上个Session
  ```bash
  # Resume the most recent CLI session
  hermes --continue
  hermes -c
  
  # Or with the chat subcommand
  hermes chat --continue
  hermes chat -c
  ```
  - 这会从 SQLite 数据库中查找最近的 cli 会话，并加载其完整的对话历史记录。

## 按名称resume
  如果您已为会话指定了标题（请参阅下文 “会话命名 ”），则可以按名称恢复会话：  
  ```bash
  # Resume a named session
  hermes -c "my project"
  
  # If there are lineage variants (my project, my project #2, my project #3),
  # this automatically resumes the most recent one
  hermes -c "my project"   # → resumes "my project #3"
  ```

## 恢复特定会话
  ```bash
  # Resume a specific session by ID
  hermes --resume 20250305_091523_a1b2c3d4
  hermes -r 20250305_091523_a1b2c3d4
  
  # Resume by title
  hermes --resume "refactoring auth"
  
  # Or with the chat subcommand
  hermes chat --resume 20250305_091523_a1b2c3d4
  ```

  退出 CLI 会话时会显示会话 ID，可以使用 hermes sessions list 找到它们。  

## 恢复对话概要
  当您恢复会话时，Hermes 会在输入提示符前以样式面板的形式显示上次对话的简要回顾：  

  恢复模式会显示一个简洁的概要面板，其中包含最近用户和助手的操作，然后再将您返回到实时提示符。  

  概要：  
  - 显示用户消息 （金色 ● ）和助手回复 （绿色 ◆ ）
  - 截断过长的消息（用户消息最多 300 个字符，助手消息最多 200 个字符/3 行）
  - 将工具调用次数合并为包含工具名称的计数（例如， [3 tool calls: terminal, web_search] ）
  - 隐藏系统消息、工具结果和内部推理
  - 仅显示最近 10 次交流，并带有“...之前的 N 条消息...”指示符
  - 使用暗色调样式来区别于正在进行的对话

  要禁用命令摘要并保持最简单行命令行为，请在 ~/.hermes/config.yaml 中进行设置：  
  ```bash
  display:
    resume_display: minimal   # default: full
  ```

  > [!IMPORTANT]  
  > 会话 ID 遵循 YYYYMMDD_HHMMSS_<hex> 的格式——CLI/TUI 会话使用 6 个字符的十六进制后缀（例如 20250305_091523_a1b2c3 ），网关会话使用 8 个字符的后缀（例如 20250305_091523_a1b2c3d4 ）。您可以通过 ID（完整或唯一前缀）或标题恢复会话——两者都支持 -c 和 -r 参数。  

# 会话命名
  给会话添加易于理解的标题，以便轻松查找和恢复会话  

## 自动生成的标题
  Hermes 会在首次对话后自动为每个会话生成一个简短的描述性标题（3-7 个字）。此过程在后台线程中使用快速辅助模型运行，因此不会增加延迟。您可以使用 hermes sessions list 或 hermes sessions browse 命令浏览会话时看到自动生成的标题。  

  自动添加标题功能每个会话只会触发一次，如果您已经手动设置了标题，则会跳过此功能。  

## 手动设置标题
  在任何聊天会话（CLI 或网关）中使用 /title 斜杠命令：  
  ```bash
  /title my research project
  ```

  标题会立即生效。如果数据库中尚未创建会话（例如，您在发送第一条消息之前运行了 /title ），则会将其放入队列，并在会话开始时应用。  

  您还可以通过命令行重命名现有会话：  
  ```bash
  hermes sessions rename 20250305_091523_a1b2c3d4 "refactoring auth module"
  ```

## 标题规则
  - 独一无二 ——任何两场会议都不能使用相同的标题。
  - 最多 100 个字符 ——保持列表输出简洁
  - 已清理 ——控制字符、零宽度字符和从右到左的字符覆盖将被自动移除。
  - 普通 Unicode 字符集没问题 ——表情符号、中日韩字符、带重音符号的字符都能正常显示。

## 自动压缩
  当会话上下文被压缩时（无论是通过 /compress 还是自动压缩），Hermes 会创建一个新的延续会话。如果原会话有标题，新会话会自动获得一个编号标题：  
  ```bash
  "my project" → "my project #2" → "my project #3"
  ```

  当您按名称恢复（ hermes -c "my project" ）时，它会自动选择谱系中的最新会话。  

## 消息平台中的标题
  /title 命令适用于所有网关平台（Telegram、Discord、Slack、WhatsApp）：  
  - /title My Research — 设置会话标题
  - /title — 显示当前标题

# 会话管理命令
  Hermes 通过 hermes sessions 提供一整套会话管理命令：  

## 列出会话
  ```bash
  # List recent sessions (default: last 20)
  hermes sessions list
  
  # Filter by platform
  hermes sessions list --source telegram
  
  # Show more sessions
  hermes sessions list --limit 50
  ```

  当会话有标题时，输出将显示标题、预览和相对时间戳：  
  ```bash
  Title                  Preview                                  Last Active   ID
  ────────────────────────────────────────────────────────────────────────────────────────────────
  refactoring auth       Help me refactor the auth module please   2h ago        20250305_091523_a
  my project #3          Can you check the test failures?          yesterday     20250304_143022_e
  —                      What's the weather in Las Vegas?          3d ago        20250303_101500_f
  ```

  如果没有会议标题，则使用更简单的格式：  
  ```bash
  Preview                                            Last Active   Src    ID
  ──────────────────────────────────────────────────────────────────────────────────────
  Help me refactor the auth module please             2h ago        cli    20250305_091523_a
  What's the weather in Las Vegas?                    3d ago        tele   20250303_101500_f
  ```

## 导出会话
  ```bash
  # Export all sessions to a JSONL file
  hermes sessions export backup.jsonl
  
  # Export sessions from a specific platform
  hermes sessions export telegram-history.jsonl --source telegram
  
  # Export a single session
  hermes sessions export session.jsonl --session-id 20250305_091523_a1b2c3d4
  ```
  - 导出的文件每行包含一个 JSON 对象，其中包含完整的会话元数据和所有消息。

## 删除会话
  ```bash
  # Delete a specific session (with confirmation)
  hermes sessions delete 20250305_091523_a1b2c3d4
  
  # Delete without confirmation
  hermes sessions delete 20250305_091523_a1b2c3d4 --yes
  ```

## 重命名会话
  ```bash
  # Set or change a session's title
  hermes sessions rename 20250305_091523_a1b2c3d4 "debugging auth flow"
  
  # Multi-word titles don't need quotes in the CLI
  hermes sessions rename 20250305_091523_a1b2c3d4 debugging auth flow
  ```
  - 如果标题已被其他会话使用，则会显示错误。

## 修剪旧会话
  ```bash
  # Delete ended sessions older than 90 days (default)
  hermes sessions prune
  
  # Custom age threshold
  hermes sessions prune --older-than 30
  
  # Only prune sessions from a specific platform
  hermes sessions prune --source telegram --older-than 60
  
  # Skip confirmation
  hermes sessions prune --older-than 30 --yes
  ```

  > [!WARNING]  
  > 清理操作只会删除已结束的会话（已明确结束或自动重置的会话）。活动会话永远不会被清理。  

## 会话统计
  ```bash
  hermes sessions stats
  ```

  输出：  
  ```bash
  Total sessions: 142
  Total messages: 3847
    cli: 89 sessions
    telegram: 38 sessions
    discord: 15 sessions
  Database size: 12.4 MB
  ```

  如需更深入的分析（例如代币使用情况、成本估算、工具细分和活动模式），请使用 hermes insights 。  

# 会话搜索工具
  该agent内置了 session_search 工具，使用 SQLite 的 FTS5 引擎对所有过去的对话执行全文搜索。  

## 工作原理
  - FTS5 搜索与相关性排名匹配的消息
  - 按会话分组结果，取前 N 个唯一会话（默认 3 个）
  - 加载每次会话的对话，截取以匹配项为中心的约 10 万个字符。
  - 发送到快速摘要模型以生成重点摘要
  - 返回包含元数据和上下文信息的会话摘要

## FTS5 查询语法
  搜索支持标准 FTS5 查询语法：  
  - 简单关键词： docker deployment
  - 短语： "exact phrase"
  - 布尔值： docker OR kubernetes ， python NOT java
  - 前缀： deploy*

## 何时使用
  系统会自动提示agent使用会话搜索：  
  ```bash
  “当用户提及过去对话中的内容，或者你怀疑存在相关的先前上下文时，请使用 session_search 来回忆起它，然后再要求他们重复一遍。”
  ```

# 按平台会话跟踪

## 网关会话
  在即时通讯平台上，会话由从消息源生成的确定性会话密钥进行加密：  
  | Chat Type | Default Key Format | Behavior |
  |-----------|--------------------|----------|
  | Telegram DM | agent:main:telegram:dm:<chat_id> | 1个DM只有1个Session。 |
  | Discord DM | agent:main:discord:dm:<chat_id> | 1个DM只有1个Session。 |
  | WhatsApp DM | agent:main:whatsapp:dm:<canonical_identifier> | 1个DM只有1个Session（当存在映射时，LID/电话别名合并为一个身份） |
  | Group chat | agent:main:<platform>:group:<chat_id>:<user_id> | 当平台公开用户 ID 时，组内的每个用户都可获取该 ID。 |
  | Group thread/topic | agent:main:<platform>:group:<chat_id>:<thread_id> | 共享会话（默认），适用于所有线程参与者。如果 thread_sessions_per_user: true ，则每个用户均可使用一个会话。 |
  | Channel | agent:main:<platform>:channel:<chat_id>:<user_id> | 当平台公开用户 ID 时，频道内的每个用户都会收到相关信息。 |

  当 Hermes 无法获取共享聊天室的参与者标识符时，它会回退到该房间的一个共享会话。  

## 共享式小组会议与独立式小组会议
  默认情况下，Hermes 在 config.yaml 中使用 group_sessions_per_user: true 。这意味着：  
  - Alice 和 Bob 可以在同一个 Discord 频道中与 Hermes 对话，而无需共享聊天记录。
  - 一个用户长时间使用大量工具的操作不会污染另一个用户的上下文窗口。
  - 中断处理也保持每个用户独立，因为运行agent密钥与隔离会话密钥一样。

  如果您想要一个共享的“房间主控系统”，请设置：  
  ```bash
  group_sessions_per_user: false
  ```
  这样就将群组/频道恢复为每个房间的单个共享会话，这既保留了共享的对话上下文，也共享了令牌成本、中断状态和上下文增长。  

## 会话重置策略
  网关会话将根据可配置策略自动重置：  
  - 空闲状态 — 不活动 N 分钟后重置
  - 每日 ——每天在特定时间重置
  - 两者都重置——以先到者为准（空闲重置或每日重置）
  - 无 — 从不自动重置

  在会话自动重置之前，agent有机会保存对话中的任何重要记忆或技能。  

  无论策略如何，具有活动后台进程的会话永远不会自动重置。  

# 存储位置
  | What | Path | 描述 |
  |------|------|------|
  | SQLite database | ~/.hermes/state.db | 所有会话元数据 + 包含 FTS5 的消息 |
  | Gateway transcripts | ~/.hermes/sessions/ | 每个会话的 JSONL 转录文本 + sessions.json 索引 |
  | Gateway index | ~/.hermes/sessions/sessions.json | 将会话密钥映射到活动会话 ID |
  
  SQLite 数据库采用 WAL 模式，支持并发读取和单个写入，非常适合网关的多平台架构。  

## 数据库架构
  state.db 中的 主要的表：  
  - sessions — 会话元数据（id、来源、user_id、模型、标题、时间戳、token 计数）。标题具有唯一索引（允许 NULL 标题，但非 NULL 标题必须唯一）。
  - messages — 完整消息历史记录（角色、内容、工具调用、工具名称、令牌计数）
  - messages_fts — 用于对消息内容进行全文搜索的 FTS5 虚拟表

# 会话过期和清理

## 自动清理
  - 网关会话根据配置的重置策略自动重置
  - 启用自动修剪：当 sessions.auto_prune 为 true 时，超过 sessions.retention_days （默认为 90 天）的已结束会话将在 CLI/网关启动时被修剪。
  - 在执行实际删除行的修剪操作后，会对 state.db 执行 VACUUM 操作以回收磁盘空间（SQLite 在执行普通 DELETE 操作时不会缩小文件）。
  - 修剪操作最多每隔 sessions.min_interval_hours （默认值 24 小时）运行一次；上次运行的时间戳记录在 state.db 中，因此同一 HERMES_HOME 目录下的所有 Hermes 进程都可以共享该时间戳。

  默认情况下已关闭 ——会话历史记录对于 session_search 调用非常重要，静默删除可能会让用户感到意外。请在 ~/.hermes/config.yaml 中启用：  

## 手动清理
  ```bash
  # Prune sessions older than 90 days
  hermes sessions prune
  
  # Delete a specific session
  hermes sessions delete <session_id>
  
  # Export before pruning (backup)
  hermes sessions export backup.jsonl
  hermes sessions prune --older-than 30 --yes
  ```

  > [IMPORTANT]  
  > 数据库增长缓慢（通常情况下，数百个会话仅占用 10-15 MB），并且会话历史记录支持 session_search 调用过去的对话，因此默认禁用自动清理功能。如果您运行的是繁重的网关/定时任务工作负载，并且 state.db 对性能有显著影响（观察到的故障模式：state.db 文件大小为 384 MB，约 1000 个会话，导致 FTS5 插入和 /resume 列表操作速度变慢），则应启用自动清理功能。对于一次性清理，请使用 hermes sessions prune 而无需启用自动清理功能。
