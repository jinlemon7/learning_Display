# 先决条件
  - uv （快速 Python 包管理器）
  - Python 3.11 （通过 uv 安装，无需 sudo）
  - Node.js v22 （用于浏览器自动化和 WhatsApp 桥接）
  - ripgrep （快速文件搜索）
  - ffmpeg （用于 TTS 的音频格式转换）

  > [!WARNING]
  > 您无需手动安装 Python、Node.js、ripgrep 或 ffmpeg。安装程序会自动检测缺失的组件并为您安装。只需确保 git 可用（ git --version ）。


# 安装 Hermes Agent
  运行一行安装程序：  
  ```bash
  # Linux / macOS / WSL2 / Android (Termux)
  curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
  ```

  完成后，重载shell：  
  ```bash
  source ~/.bashrc   # or source ~/.zshrc
  ```

# 最快路径
  选择符合您目标的行：  
  | 目标 | 先做这件事 | 然后这样做 |  
  |------|------------|------------|  
  | 我只想让 Hermes 在我的机器上运行。 | 运行 `hermes setup` | 运行一次真实的聊天测试，并验证其响应情况。 |  
  | 我已经知道我的服务提供商了。 | 运行 `hermes model` | 保存配置，然后开始聊天。 |  
  | 我想要一个机器人或始终在线的设置。 | 在 CLI 成功设置 Hermes 后，运行 `hermes gateway setup` | 连接 Telegram、Discord、Slack 或其他平台。 |  
  | 我想要一个本地或自托管的模型。 | 运行 `hermes model` 并选择自定义端点 | 验证端点、模型名称和上下文长度。 |  
  | 我想要多提供商fallback。 | 首先运行 `hermes model` | 只有在基本聊天功能正常运行后，才添加路由和fallback机制。 |  

  经验法则： 如果 Hermes 无法完成正常的聊天，就不要添加更多功能。先确保能够流畅地进行对话，然后再逐步添加网关、定时任务、技能、语音或路由等功能。

  | AWS Bedrock | 通过原生 Converse API 使用 Claude、Nova、Llama、DeepSeek | IAM 角色或运行 `aws configure`（参考指南） |
  | Kilo Code | KiloCode 托管模型 | 设置 `KILOCODE_API_KEY` |
  | OpenCode Zen | 按需付费访问精选模型 | 设置 `OPENCODE_ZEN_API_KEY` |

  > [!WARNING]
  > Hermes Agent 要求模型至少包含 64,000 个上下文标记 。窗口较小的模型无法为多步骤工具调用工作流程维护足够的内存，因此会在启动时被拒绝。大多数托管模型（例如 Claude、GPT、Gemini、Qwen 和 DeepSeek）都能轻松满足此要求。如果您运行的是本地模型，请将其上下文大小设置为至少 64K

# 设置如何存储
  Hermes 将密钥与普通配置分开：  
  - secret 和 token → ~/.hermes/.env
  - 非机密设置 → ~/.hermes/config.yaml

  设置值的最简单方法是通过命令行界面 (CLI)：  
  ```bash
  hermes config set model anthropic/claude-opus-4.6
  hermes config set terminal.backend docker
  hermes config set OPENROUTER_API_KEY sk-or-...
  ```
  - 正确的值会自动写入正确的文件。


# 验证会话是否正常工作
  在继续之前，请确保resume功能正常：  
  ```bash
  hermes --continue    # Resume the most recent session
  hermes -c            # Short form
  ```

  这样应该能让你回到刚才的会话。如果不行，请检查你是否在同一个用户配置文件中，以及会话是否已保存。这一点在你以后管理多个设置或机器时很重要。

# 常见故障模式
  以下是一些最浪费时间的问题：  
  | 症状 | 可能原因 | 修复 |
  |------|----------|------|
  | Hermes打开了信箱，但只给了空洞或不完整的回复。 | 提供商身份验证或型号选择错误 | 再次运行 `hermes model` 并确认提供商、模型和身份验证。 |
  | 自定义端点“有效”，但返回垃圾数据。 | 基础 URL、模型名称错误，或者实际上与 OpenAI 不兼容。 | 首先在另一个客户端中验证端点。 |
  | 网关已启动，但无人可以向其发送消息。 | 机器人token、白名单或平台设置不完整 | 重新运行 `hermes gateway setup` 并检查 `hermes gateway status`。 |
  | `hermes --continue` 找不到旧会话 | 切换配置文件或会话未保存 | 查看 `hermes sessions list` 并确认您登录的是正确的个人资料。 |
  | 模型不可用或出现异常的备用行为 | 提供商路由或fallback设置过于激进。 | 保持路由关闭状态，直到基础服务提供商稳定为止。 |
  | `hermes doctor` 标志配置问题 | 配置值缺失或已过期。 | 修复配置，在添加功能之前重新测试一下简单的聊天功能。 |

# 安装程序的功能
  安装程序会自动处理所有事项——所有依赖项（Python、Node.js、ripgrep、ffmpeg）、仓库克隆、虚拟环境、全局 hermes 命令设置以及 LLM 提供程序配置。安装完成后，即可开始聊天。  

## 安装布局
  安装程序将文件放置的位置取决于您是以普通用户身份还是以 root 用户身份进行安装：  
  | 安装程序 | 代码存在于 | hermes 二进制 | 数据目录 |
  |----------|------------|---------------|----------|
  | 按用户（正常） | ~/.hermes/hermes-agent/ | ~/.local/bin/hermes （符号链接） | ~/.hermes/ |
  | 以 root 用户身份运行（ sudo curl … | sudo bash ） | /usr/local/lib/hermes-agent/ | /usr/local/bin/hermes | /root/.hermes/ （或 $HERMES_HOME） |

  根模式 FHS 布局 （ /usr/local/lib/… ， /usr/local/bin/hermes ）与 Linux 上其他系统级开发工具的安装位置一致。这对于共享机器部署非常有用，因为一个系统安装应该服务于所有用户。每个用户的配置（身份验证、技能、会话）仍然位于每个用户的 ~/.hermes/ 下或显式指定的 HERMES_HOME 下。  

# 更新
  只需一条命令即可更新到最新版本：  
  ```bash
  hermes update
  ```
  - 这会拉取最新代码，更新依赖项，并提示您配置自上次更新以来添加的任何新选项。

  > [!IMPORTAMT]
  > hermes update 会自动检测新的配置选项并提示您添加。如果您跳过了该提示，可以手动运行 hermes config check 查看缺失的选项，然后 hermes config migrate 以交互方式添加它们。

# 更新过程中会发生什么
  1. 配对数据快照 — 保存一个轻量级的更新前状态快照（涵盖 ~/.hermes/pairing/ 、Feishu 注释规则以及其他在运行时修改的状态文件）。可通过 hermes backup restore --state pre-update 回滚。
  2. Git pull—— 从 main 分支拉取最新代码并更新子模块
  3. 安装依赖项 ——运行 uv pip install -e ".[all]" 来安装新增或已更改的依赖项。
  4. 配置迁移 ——检测自您当前版本以来新增的配置选项，并提示您进行设置。
  5. 网关自动重启 ——更新完成后，正在运行的网关会自动刷新，以便新代码立即生效。由服务管理的网关（Linux 系统使用 systemd，macOS 系统使用 launchd）通过服务管理器重启。手动启动的网关会在 Hermes 能够将运行中的进程 ID 映射回配置文件时自动重新启动。

# 仅供预览：hermes update --chec
  想知道在实际拉取更新之前是否落后于 origin/main 吗？运行 hermes update --check 即可。它会获取更新，并将本地提交和最新的远程提交并排打印出来，如果同步则返回 0 ，如果落后则返回 1 此操作不会修改任何文件，也不会重启网关。在需要判断“是否有更新”的脚本和定时任务中非常有用。  

# 更新前完整备份：--backup
  对于高价值配置文件（生产网关、共享团队安装），您可以选择对 HERMES_HOME （配置、身份验证、会话、技能、配对）进行完整的预拉取备份：  
  ```bash
  hermes update --backup
  ```

  或者将其设置为每次运行的默认值：  
  ```bash
  # ~/.hermes/config.yaml
  update:
    backup: true
  ```

  预期输出结果如下：  
  ```bash
  $ hermes update
  Updating Hermes Agent...
  📥 Pulling latest code...
  Already up to date.  (or: Updating abc1234..def5678)
  📦 Updating dependencies...
  ✅ Dependencies updated
  🔍 Checking for new config options...
  ✅ Config is up to date  (or: Found 2 new options — running migration...)
  🔄 Restarting gateways...
  ✅ Gateway restarted
  ✅ Hermes Agent updated successfully!
  ```

# 建议进行更新后验证
  hermes update 处理主要的更新路径，但快速验证确认所有内容都已顺利完成：  
  1. git status --short — 如果目录树意外变脏，请在继续操作前进行检查
  2. hermes doctor 检查配置、依赖项和服务健康状况
  3. hermes --version — 确认版本号已按预期更新
  4. 如果您使用网关： hermes gateway status
  5. 如果 doctor 报告 npm audit 问题：在标记的目录中运行 npm audit fix

  > [!WARNING]
  > 更新后的工作树混乱
  > 如果在 hermes update 后， git status --short 显示意外更改，请停止操作并检查这些更改后再继续。这通常意味着本地修改被重新应用到更新后的代码之上，或者某个依赖项步骤刷新了锁定文件。

# 如果您的终端在更新过程中断开连接
  hermes update 可防止意外丢失：  
  - 此次更新会忽略 SIGHUP ，因此关闭 SSH 会话或终端窗口不再会导致安装过程中进程中断。pip 和 git 子进程也会继承此保护机制，因此即使连接断开 pip Python 环境也不会出现安装不完整的情况。
  - 更新运行时，所有输出都会镜像到 ~/.hermes/logs/update.log 中。如果您的终端消失，请重新连接并检查日志，以查看更新是否完成以及网关重启是否成功：
    ```bash
    tail -f ~/.hermes/logs/update.log
    ```
  - Ctrl-C （SIGINT）和系统关机（SIGTERM）仍然有效——这些都是故意的取消，而不是意外。

# 回滚指令
  如果更新引入了问题，您可以回滚到以前的版本：  
  ```bash
  cd /path/to/hermes-agent
  
  # List recent versions
  git log --oneline -10
  
  # Roll back to a specific commit
  git checkout <commit-hash>
  git submodule update --init --recursive
  uv pip install -e ".[all]"
  
  # Restart the gateway if running
  hermes gateway restart
  ```

  要回滚到特定版本标签：  
  ```bash
  git checkout v0.6.0
  git submodule update --init --recursive
  uv pip install -e ".[all]"
  ```

  > [!WARNING]
  > 如果添加了新选项，回滚可能会导致配置不兼容。回滚后运行 hermes config check ，如果遇到错误，请从 config.yaml 中删除任何无法识别的选项。
