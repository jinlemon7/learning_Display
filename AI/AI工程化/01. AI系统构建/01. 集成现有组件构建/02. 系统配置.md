# 系统目标配置
  在程序员工作量和AI开发质量之间寻找极致的平衡。  

# AI组件运行原则配置
  [andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)中的四大原则：  
  - 先想后码（Think Before Coding）
  - 极简优先（Simplicity First）
  - 精准修改（Surgical Changes）
  - 目标驱动（Goal-Driven Execution）

# 系统架构和功能配置

## 核心原则
  在给出具体配置前，我们先要转变一个根本的设计思路：让 OpenClaw 更像一个表达“意图”架构师，而 Hermes 集群则像一个严格遵循工程纪律的工匠团队。

  - OpenClaw (指挥官)：负责表达“做什么”和“为什么”，而不是“怎么做”。
  - Hermes (实干家)：负责在严格的约束下，以最小的代价、最精准的方式达成目标，并自我验证。

## OpenClaw 指挥官与网关：从“动作序列”到“目标树”

### 任务拆解的粒度：拆解为“可验证的子目标”
  为了对齐目标驱动和先想后码，任务拆解不应输出“动作序列”，而应输出一颗目标树。  
  - 推荐配置：OpenClaw 将“为新服务创建支付功能”这类自然语言指令，拆解为：
    ```text
    [假设] 我们将使用已审批通过的 [Stripe] 和 [RESTful 风格]。
    目标: 实现支付服务
      |- 子目标: 用户可创建支付订单 (通过测试验证)
      |- 子目标: 正确处理 Stripe Webhook (通过mock Webhook测试验证)
      |- 子目标: 支付失败时有明确错误码 (通过测试验证)
    风险和取舍: [Stripe] 相比 [XX支付] 开发快但抽成高，是否接受？
    ```

### 多 Agent 协同调度：基于“契约”而非“时间”
  告别复杂的任务DAG，改用更简单的契约驱动。  
  - 推荐配置：
    - “订单服务Agent A”和“支付服务Agent B”同时启动。它们的首要任务是共同定义一份API契约（如一个OpenAPI文件）。
    - 契约完成后，两个Agent便完全解耦。Agent A作为客户端，只需生成符合契约的Mock，与Agent B的真实接口并行开发。
    - OpenClaw的调度极其简单：先派发“谈判制定契约”的任务，契约文件提交到仓库后，再派发各自的实现任务。

### 网关的边界：仅作路由，拒绝协议转换
  遵循极简优先原则，网关不引入复杂逻辑。  
  - 推荐配置：
    - 外部API网关：仅负责接收外部Webhook等事件，进行认证/授权，然后原封不动地转发给内部消息队列。不负责解析和转换，避免端到端的调试困难。
    - 内部Agent通信枢纽：极简的消息队列（Topic），拒绝花哨的广播/路由。遵循精准修改原则，Agent只订阅自己需要的Topic。
   
## Hermes 实干专家的内嵌纪律
### Agent 的角色定义：“技能即约束”
  Agent的能力是“一个核心规则 + 一组技能约束包”。  
  推荐配置：  
  一个“支付Agent”其实是一个普通的 Agent-Generic，加载了 skill-pack-payment.yaml：  
  ```yaml
  name: "支付技能包"
  constraints: # 核心是约束
    - pattern: "API设计"
      rule: "必须保持与项目中其他API一致的风格，不要自造风格" # Surgical Changes
    - pattern: "错误处理"
      rule: "复用 src/utils/errors.ts 中的 AppError，不要引入新的错误类" # Simplicity First
  ```
  - 这样能杜绝Agent为了“灵活性”而引入新抽象。

### ACP 协议/桥接工具：强制对齐的“握手”
  ACP的核心任务是强制Agent先验证意图再执行，落实“先想后码”。  
  - 推荐配置：桥接工具接入“数据库”时，不允许Agent直接发SQL。其流程必须是：
    1. Agent声明意图：“我需要更新 orders 表中 ID 为 X 的 status 字段。”
    2. 工具返回当前表结构、索引和关联关系。
    3. Agent据此生成待执行的SQL。
    4. Agent自问：“这个 UPDATE 会影响哪些索引？会不会锁表？有没有更好的写法？”
    5. 最后执行。
   
    最后执行。

### 实干执行时的自主权：绝对自主，但有“护栏”
  Agent获得完全的“怎么做”自主权，但其行动空间被“护栏”约束。  
  推荐配置：  
  - 方向护栏：遇到不在目标树内的“诱惑”（如觉得某个类该重构），它必须先在日志中提出建议，然后专注当前任务，绝不能动手。
  - 安全护栏：Agent的沙箱环境里没有 git push --force 等命令；代码生成后必须通过“门禁”（lint、安全检查），否则无法进入测试阶段。

## 持续学习与进化：基于目标的反馈闭环
### 经验沉淀的颗粒度：沉淀“目标达成报告”
  不再是零散的代码片段，而是沉淀整个目标达成报告。  
  - 推荐配置：每次任务结束后，Hermes必须输出一个包含以下内容的报告：  
    - 目标：最初拆解的子目标。
    - 假设与权衡：验证了哪些假设？选择了什么方案及原因？
    - 最终产出：关键代码、测试、API文档的链接。
    - 遗留问题：故意留下了哪些“设计债”或未清理的冗余。
   
  这份报告就是技能库的核心资产。  

### 反馈优化技能的途径：优化“思考过程”
  - 推荐配置：如果一个Agent过度设计，我们会这样做
    - 给这个“解决模式”标记失败。
    - 复盘：Agent违反了“极简优先”原则。
    - 修复：在该Agent的技能包里追加一条新的约束规则：禁止引入抽象层，直到同一模式出现3次以上。
   
  这条新规则会成为后续Agent的强制记忆。  


### 自动进化的安全边界：无QA，不进化的规范
  “无规范”本身就是最强防线。  
  - 推荐配置：技能库中的任何约束规则更新，都必须以一个因违反该规则而导致失败的案例为前提。任何规则的新增或修改，都必须像代码一样，以PR的形式提交，并附上案例证据，由Senior Developer评审后才能合并。这坚决杜绝普遍存在的的黑箱式自动进化。

### 基础设施的极简与精准

###  Git 仓库：一份PR，一个目
  极致对齐精准修改原则。  
  - 推荐配置：一个Agent任务结束后，其PR必须只包含与目标直接相关的更改。CI系统会强制进行 “Diff相关性检查” ，拒绝包含无关“顺手重构”的PR。同时，只允许Agent清理自身改动造成的孤儿代码，不能删除任何预先存在的冗余。

### CI/CD 流水线：目标驱动的“一路向左”
  流水线是“目标驱动”的最终验证者。  

  - 推荐配置：
    - 触发：任何代码推送都触发CI/CD，不准设置例外。
    - 流程：执行与本次目标相关的所有测试。测试必须100%通过，不允许存在“已知失败”的测试，若有则Agent必须先修复。
    - 部署：Dev/Staging环境无人工干预的全自动部署。生产环境部署由测试报告驱动的“发布按钮”触发。
   
###  容器镜像与部署：重申不造轮子
  - 推荐配置：硬性规定Agent不要尝试发明新的部署方式。生成Dockerfile必须继承项目指定的官方基础镜像，应用配置必须沿用现有ConfigMap/Secret的方案。

## 用户交互与系统边界

### 用户如何参与：用户即“领域专家”
  对齐先想后码和目标驱动。  
  - 推荐配置：用户只在三个关键节点介入：
    1. 对齐目标时：确认OpenClaw拆解的目标树、假设和风险，明确“做什么”。
    2. 消除歧义时：Hermes执行中遇到假设之外的歧义，会带着方案A/B主动提问，而不是静默猜测。
    3. 验收结果时：审查并合并PR，这是唯一需要审阅代码的节点，核心审察代码是否有越界改动。
   
### 多用户与多项目支持
  - 推荐配置：全局技能库（通用约束）和项目技能库（项目级约束）彻底分离。各项目仓库严格隔离，理想情况下每个项目运行独立的低权限CI Runner。

### 非功能需求
  - 可观测性：门禁指标聚焦“目标完成率”、“平均PR大小”（衡量精准度）、“因违反原则被拒绝的PR比例”。
  - 部署形态：核心引擎的本地/私有化部署作为唯一选项，以确保对企业代码资产的绝对控制。

## 总结
  总的来说，这套配置通过 目标树、契约驱动、强制对齐的握手、约束即技能、目标达成报告 等一系列机制，将四大原则融入系统骨架，最终形成一个能用工程纪律约束自身，并基于清晰目标持续进化的虚拟开发团队。
