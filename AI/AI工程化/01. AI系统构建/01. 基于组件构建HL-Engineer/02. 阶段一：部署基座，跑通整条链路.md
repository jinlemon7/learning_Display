# 顶层部署层次图
  ```text
  ~/hl-engineer/                    # ★ HL‑Engineer 系统根目录，所有相关文件集中于此
  ├── bin/                           # 自定义可执行脚本（启动/停止/同步）
  ├── config/                        # 组件配置文件（从默认位置链接或拷贝而来）
  │   ├── openclaw/                    # OpenClaw 配置（等效 ~/.openclaw）
  │   └── hermes/                      # Hermes 配置（等效 ~/.hermes）
  ├── data/                          # 运行时产生的数据（技能、规则、进化产物）
  │   ├── openclaw/                    # OpenClaw 的技能、规则存放处
  │   │   ├── rules/                     # 四大原则等 rules
  │   │   └── skills/                    # 本地技能库
  │   ├── hermes/                      # Hermes 沉淀的技能、历史对话
  │   │   └── skills/                    # 进化出的 agentskills.io 标准技能
  │   └── acp/                         # ACP 桥接状态文件、任务队列等
  ├── logs/                          # 所有日志集中存放
  │   ├── openclaw.log                 # OpenClaw 后台日志
  │   ├── hermes.log                   # Hermes 后台日志
  │   └── acp-bridge.log               # ACP 桥接服务日志
  ├── projects/                      # 测试/生产 Git 仓库（Hermes 的工作区）
  │   └── test-hl-engineer/            # 示例测试仓库（后续阶段二会用到）
  ├── acp-bridge/                    # ACP 桥接服务代码（阶段二创建，但目录先预留）
  ├── scripts/                       # 辅助脚本（比如阶段三的技能同步脚本）
  └── README.md                      # 整体说明
  ```

  重要文件和目录详细注释：  
  - `~/hl-engineer/bin/`：存放所有自制管理脚本，比如 start-all.sh、stop-all.sh、sync-skills.sh，赋予系统级的可执行权限，便于日常运维。
  - config/openclaw/：计划用软链接替换默认的 ~/.openclaw，这样任何对 OpenClaw 配置的修改都会发生在统一目录中，备份时只备份 ~/hl-engineer 即可。
  - data/openclaw/rules/：存放阶段二要注入的《四大原则》，文件名例如 hl-engineer-principles.md。
  - data/openclaw/skills/：存放 OpenClaw 本地技能，阶段二会创建 delegate-to-hermes.md，阶段三会通过 data/hermes/skills/ 同步进化技能过来。
  - data/hermes/skills/：Hermes 完成任务后自动生成的 agentskills.io 标准技能文件，由定时脚本 sync-skills.sh 单向同步到 data/openclaw/skills/。
  - logs/：统一收集所有日志，方便调试整个系统链路。
  - projects/：Hermes 进行代码生成和提交的 Git 仓库根目录，避免与个人其他项目混淆。

# 卸载并重装WSL2（Windows Subsystem for Linux 2）

## 注销所有已安装的 WSL 发行版
  ```shell
  wsl -l -v
  ```

  然后，逐个注销并删除它们：  
  ```shell
  wsl --unregister <发行版名称>
  ```

## 移除 WSL 核心组件与数据
  ```shell
  wsl --uninstall
  ```
  - 该命令会自动卸载 WSL 组件并删除大部分数据。完成后重启电脑。

## 全新安装 WSL2
  ```shell
  # 一键安装 WSL2 及默认 Ubuntu 发行版
  wsl --install

  # 如需安装特定版本（如 Ubuntu-24.04），改为：
  wsl --install -d Ubuntu-24.04
  ```
  - 若提示已禁止403：开了代理，需要换成直连或者关掉代理。

  shell下载WSL2非常慢，使用手动安装方式：  
  [https://github.com/microsoft/WSL/releases](https://github.com/microsoft/WSL/releases)查找最新版本的wsl.xxxx.x64.msi安装包  
  然后安装Ubuntu 24.04系统：  
  ```bash
  # 列出可用的Linux系统分发
  wsl.exe --list --online

  # 使用“wsl.exe --install <Distro>”安装
  wsl.exe --install Ubuntu-24.04
  ```

## 设置WSL2
  启动方式：  
  - 使用shell命令：`wsl.exe -d Ubuntu-24.04`
  - 点击win开始菜单添加的“Ubuntu 24.04”图标
  - 安装完成后自动启动Ubuntu 24.04

  第一次启动会提示创建root用户和密码  

# 安装 Docker
  ```bash
  # 下载并运行 Docker 官方安装脚本
  curl -fsSL https://get.docker.com |sudo sh
  
  # 将当前用户加入 docker 组，避免每次使用 sudo
  sudo usermod -aG docker $USER
  
  # 重新登录或刷新用户组
  newgrp docker
  ```

## 配置 Docker
  为了更顺畅地拉取镜像，可以配置国内镜像加速器。我用阿里云加速器来示范，你也可以使用腾讯云、中科大等其他源。  
  创建 Docker 配置文件：  
  ```bash
  sudo mkdir -p /etc/docker
  ```

  创建并编辑 daemon.json 文件：  
  ```bash
  sudo tee /etc/docker/daemon.json <<'EOF'
  {
    "registry-mirrors": [
      "https://mirrors.aliyun.com",
      "https://docker.mirrors.ustc.edu.cn",
      "https://hub-mirror.c.163.com",

      "https://docker.1panel.live",
      "https://hub.rat.dev",
      "https://docker.m.daocloud.io"
    ]
  }
  EOF
  ```

  重启 Docker 使配置生效：  
  ```bash
  sudo systemctl daemon-reload
  sudo systemctl restart docker
  ```

# 第一阶段：清理旧组件
  请按顺序执行以下命令，确保环境干净。  

## 备份 OpenClaw 数据
  OpenClaw 的所有配置、API 密钥和会话数据默认存储在 ~/.openclaw/ 目录下。建议运行：  
  ```bash
  cp -r ~/.openclaw ~/.openclaw.backup.$(date +%Y%m%d)
  ```
  - 如需保留自定义 Skills、密钥等，请备份目录下的 .env 和 skills/ 子目录。

## 彻底卸载 OpenClaw

### 第一步：停止并卸载守护进程
  OpenClaw 会在系统中注册持久化后台服务（systemd 用户服务），若残留，重启后会自动复活。  
  先尝试优雅停止：  
  ```bash
  openclaw gateway stop
  openclaw gateway uninstall
  ```

  若上述命令无效，手动清理 systemd 服务：  
  ```bash
  systemctl --user stop openclaw-gateway.service
  systemctl --user disable openclaw-gateway.service
  rm -f ~/.config/systemd/user/openclaw-gateway.service
  systemctl --user daemon-reload
  ```

### 第二步：执行官方卸载命令（核心）
  ```bash
  openclaw uninstall --all --yes --non-interactive
  ```

  这将删除服务、配置、数据库等全部本地组件。  
  若 CLI 命令已不可用，请直接跳至手动删除阶段。  

### 第三步：手动删除配置目录
  ```bash
  rm -rf ~/.openclaw
  ```
  - 这是最关键的一步，该目录包含凭证、API 密钥、会话记忆和 Skills，确保无残留数据遗留。

### 第四步：卸载 CLI 工具
  若通过 npm 安装，请执行：  
  ```bash
  npm uninstall -g openclaw
  ```

  验证是否已移除：  
  ```bash
  which openclaw          # 应无输出
  openclaw --version      # 应报 "command not found"
  ```

### 撤销 OAuth 令牌（安全必需）
  OpenClaw 依赖长效 OAuth 令牌连接 Google、Slack、Discord、GitHub 等服务，令牌存储在外部的云端，须手动撤销。  
  请在对应外部服务中回收相关授权（建议全程检查），否则即便本地完全清理，外部令牌仍然有效，存在持续的安全隐患。  

### 验证 OpenClaw 卸载干净
  ```bash
  [ -d ~/.openclaw ] && echo "❌ 配置目录仍存在" || echo "✅ 配置目录已清除"
  systemctl --user status openclaw-gateway.service  # 应报 "Unit not found"
  lsof -i :18789           # 检查默认端口 18789 是否释放
  ```

## 彻底卸载 Hermes

### 第一步：停止后台服务与运行时
  若存在 systemd 用户服务，请一并清理：  
  ```bash
  hermes gateway stop      # 优雅停止
  systemctl --user disable hermes-gateway        # 若适用
  rm -f ~/.config/systemd/user/hermes-gateway.service
  ```

  清理运行时目录：  
  ```bash
  rm -rf ~/.hermes/run/ ~/.hermes/logs/ ~/.hermes/gateway/sessions/
  ```

### 第二步：使用自带卸载命令
  ```bash
  hermes uninstall
  ```
  - 按提示选择是否保留 ~/.hermes/ 配置。

### 第三步：手动删除核心配置与数据目录
  ```bash
  rm -rf ~/.hermes/                      # 主配置根目录
  sudo rm -rf /etc/hermes/               # 检查并删除全局配置
  ```

### 第四步：删除 CLI 工具与包依赖
  若通过 Python 包管理器安装，执行：  
  ```bash
  pip uninstall hermes-agent -y
  ```

  验证并删除二进制：  
  ```bash
  which hermes && rm -f $(which hermes)  # 删除残留可执行文件
  ```

### 第五步：清除环境变量与 Shell 初始化项
  防止残留 HERMES_HOME 或 PATH 路径导致后续冲突：  
  ```bash
  grep -n "hermes\|HERMES" ~/.bashrc ~/.zshrc ~/.profile 2>/dev/null
  ```

  手动编辑对应配置文件，删除含 export HERMES_HOME=、export PATH=.*hermes 的行，然后运行：  
  ```bash
  source ~/.bashrc
  env | grep HERMES          # 应无输出
  ```

### 验证 Hermes 卸载干净
  ```bash
  which hermes               # 应无输出
  hermes --version           # 应报 "command not found"
  [ -d ~/.hermes ] && echo "❌ 配置目录仍存在" || echo "✅ 配置目录已清除"
  ```

## 部署 SearXNG

### 使用 Docker 部署 SearXNG
  使用 docker-compose, 此方案适合长期使用，配置和数据会持久化在宿主机，更便于管理。  

  使用以下命令生成一个随机密钥，并复制到docker-compose.yml 文件中：
  ```bash
  openssl rand -base64 32
  ```

  新建项目目录和 docker-compose.yml 文件  
  ```bash
  mkdir -p ~/searxng && cd ~/searxng
  nano docker-compose.yml
  ```

  将以下完整的配置内容复制并粘贴到 nano 编辑器中：  
  ```bash
  services:
    # Redis 缓存服务
    redis:
    image: redis:7.2-alpine
    container_name: searxng-redis
    restart: unless-stopped
    networks:
      - searxng-network
    volumes:
      - ./redis-data:/data
  
    # SearXNG 核心搜索服务
    searxng:
      image: searxng/searxng:latest
      container_name: searxng
      restart: unless-stopped
      ports:
        - "8080:8080"
      depends_on:
        - redis
      networks:
        - searxng-network
      volumes:
        - ./searxng/settings.yml:/etc/searxng/settings.yml:ro
        # 此项需确保/etc/searxng/limiter是文件而非目录，否则SearXNG启动后的状态会持续ReTrying，无法成功UP
        - ./searxng/limiter.toml:/etc/searxng/limiter.toml:ro
      environment:
        - SEARXNG_SECRET_KEY={具体的随机密钥}
        - SEARXNG_BASE_URL=http://localhost:8080
        - SEARXNG_REDIS_URL=redis://redis:6379/0
  
  networks:
    searxng-network:
      driver: bridge
  ```

  创建工作目录和配置文件  
  - 创建配置文件目录：mkdir -p searxng
  - 创建 settings.yml 配置文件并写入内容：
    确保 search.formats 中包含 json，这是后续被 OpenClaw 调用的关键。  
    ```bash
    cat > searxng/settings.yml << 'EOF'
    use_default_settings: true
    search:
      formats:
        - html
        - json
    server:
      limiter: false
    EOF
    ```

  启动服务  
  ```bash
  # 启动容器 / 重启容器
  sudo docker-compose up -d

  # 修改配置文件前，关闭运行中容器
  sudo docker-compose down
  ```

  检查是否启动成功和排查错误  
  ```bash
  # 查看容器状态：
  sudo docker-compose ps

  # 查看最后几条日志
  sudo docker logs searxng --tail 20
  ```

### 验证 SearXNG 是否正常
  测试 JSON API：  
  ```bash
  curl "http://localhost:8080/search?q=test&format=json"
  ```
  - 如果返回一段 JSON 格式的搜索结果数据，说明 SearXNG 已正常运行，并且 JSON API 也是可用的。

  ```text
  └  引导完成。请使用上面的仪表盘链接来控制 OpenClaw。
  ```

  向导完成后，立即将生成的配置和数据迁移到规划目录：  
  ```bash
  # 移动 .openclaw 目录到 ~/hl-engineer/config/openclaw
  mv ~/.openclaw ~/hl-engineer/config/openclaw
  
  # 创建软链接，让 OpenClaw 仍然认为配置在 ~/.openclaw
  ln -s ~/hl-engineer/config/openclaw ~/.openclaw
  ```

  这样，任何对 ~/.openclaw 的读写都会实际落在 ~/hl-engineer/config/openclaw 里，配置已统一。  

## 设置日志路径（可选）
  如果 OpenClaw 允许指定日志路径，可后续在 config/openclaw/config.yml 中修改。默认它可能使用 ~/.openclaw/logs，因为我们已经软链接了，所以日志会写入 ~/hl-engineer/config/openclaw/logs，我们还可以在 ~/hl-engineer/logs/openclaw.log 创建个软链接。  
  ```bash
  # 1. 确保集中日志目录存在
  mkdir -p ~/hl-engineer/logs

  # 2. 确保 config/openclaw/logs/ 目录存在（用于放置软链接）
  mkdir -p ~/hl-engineer/config/openclaw/logs

  # 3. 如果旧日志文件已经存在于 config/openclaw/logs/ 下，先移动到 logs/
  mv ~/hl-engineer/config/openclaw/logs ~/hl-engineer/logs 2>/dev/null || true

  # 4. 创建软链接：让 config 下的路径指向集中日志文件（方向正确）
  ln -s ~/hl-engineer/logs ~/hl-engineer/config/openclaw/logs
  ```

### 运行配置向导
  ```bash
  hermes setup
  ```

  向导中的关键选项：  
  - 设置方式：Quick setup
  - 推理提供商：DeepSeek
  - 默认模型：deepseek-v4-flash
  - 消息平台：QQ Bot，扫码添加
  - 安全：启用私信配对，将你的 QQ 号加入允许列表并设为主频道
  - 后台服务：选择将网关作为用户级 systemd 服务安装

  更新系统的“环境变量PATH”，使hermes命令生效  
  ```bash
  # 如果下面的命令输出了类似 /home/jinlemon7/.local/bin/hermes 的路径，就说明hermes安装成功了，只是PATH没配好
  ls -l ~/.local/bin/hermes
  which hermes

  # 认你的Shell类型：/bin/bash → 你是 bash 用户，/bin/zsh → 你是 zsh 用户
  echo $SHELL

  # 把hermes所在的目录（~/.local/bin）添加到系统的PATH里
  # 如果你用的是 bash
  echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
  source ~/.bashrc

  # 检查hermes命令是否生效
  hermes doctor
  ```
