# 自动配置原理（Auto-Configuration）
## 核心脉络: 从 @SpringBootApplication 说起
   
  ```java
  @SpringBootApplication   // 这个注解=三个注解的合成
  public class Application {
      public static void main(String[] args) {
          SpringApplication.run(Application.class, args);
      }
  }
  @SpringBootApplication = @EnableAutoConfiguration + @ComponentScan + @SpringBootConfiguration
  ```

  最关键的是 @EnableAutoConfiguration，这是自动配置的入口开关。  

## 底层实现：SpringFactoriesLoader 机制
### 步骤一：读取 spring.factories  
  当你标注 @EnableAutoConfiguration 后，Spring Boot 会加载 所有 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件。  

  这个文件里写的是全限定类名列表：  
  ```text
  org.springframework.boot.autoconfigure.web.servlet.DispatcherServletAutoConfiguration
  org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
  org.springframework.boot.autoconfigure.data.redis.RedisAutoConfiguration
  ...
  ```
  
### 步骤二：条件评估（Conditional Evaluation）
  这 130+ 个自动配置类并不会全部生效——每个配置类上都有 @Conditional 系列注解"守卫"：  
  ```java
  @Configuration                         // 这是一个配置类
  @ConditionalOnClass(DataSource.class)  // 只有当 classpath 中有 DataSource 才加载
  @ConditionalOnMissingBean(...)         // 只有当用户没有自定义 DataSource Bean 才加载
  @EnableConfigurationProperties(DataSourceProperties.class) // 绑定配置属性
  public class DataSourceAutoConfiguration {
      
      @Bean
      @ConditionalOnMissingBean
      public DataSource dataSource() {
          // 创建一个默认的 HikariCP 数据源
          return dataSourceProperties.initializeDataSourceBuilder().build();
      }
  }
  ```

  完整条件注解家族：  
  | 注解 | 含义 | 使用场景 |
  |------|------|----------|
  | @ConditionalOnClass | 类存在时生效 | 检测 classpath 有没有某个依赖 |
  | @ConditionalOnMissingClass | 类不存在时生效 | 没有某个依赖时的降级配置 |
  | @ConditionalOnBean | 容器中有某个 Bean 时生效 | 用户已配置时不再重复配置 |
  | @ConditionalOnMissingBean | 容器中没有某个 Bean 时生效 | 核心"用户优先"机制 |
  | @ConditionalOnProperty | 配置项存在/为某值时生效 | 通过配置开关启用/禁用功能 |
  | @ConditionalOnResource | 资源文件存在时生效 | 检测 classpath 中是否有某文件 |
  | @ConditionalOnWebApplication | 是 Web 应用时生效 | 区分 Web 和非 Web 场景 |
  | @ConditionalOnExpression | SpEL 表达式为 true 时生效 | 自定义复杂条件 |

### 步骤三：条件评估引擎
  Spring Boot 在启动时会创建一个 ConditionEvaluationReport，记录每个自动配置类的匹配结果：  
  - 匹配（matched）：条件全部满足，配置生效
  - 不匹配（missed）：某个条件不满足，配置被跳过

  这就是为什么启动日志里有这一堆 Positive/Negative matches：  
  
  ```bash
  ============================
  CONDITIONS EVALUATION REPORT
  ============================
  Positive matches:
  -----------------
     DataSourceAutoConfiguration matched:
        - @ConditionalOnClass found required class 'javax.sql.DataSource' (OnClassCondition)
        - @ConditionalOnMissingBean (types: javax.sql.DataSource; SearchStrategy: all) found no beans (OnBeanCondition)
  
  Negative matches:
  -----------------
     ActiveMQAutoConfiguration:
        Did not match:
           - @ConditionalOnClass did not find required class 'javax.jms.ConnectionFactory' (OnClassCondition)
  ```

## "用户优先"原则
  这是自动配置的精髓——如果用户自己声明了某个 Bean，自动配置的就不会覆盖：  
  ```java
  // 用户自定义数据源 → 自动配置的 DataSourceAutoConfiguration 会检测到 @ConditionalOnMissingBean 不满足，跳过
  @Bean
  public DataSource myDataSource() {
      return new DruidDataSource();  // 用户选了 Druid，不用 HikariCP
  }
  ```

  实现方式：自动配置类上的 @ConditionalOnMissingBean 会在容器中搜索，发现同类型 Bean 已存在则跳过。  

## 面试话术（30秒版）
  Spring Boot 自动配置基于 @EnableAutoConfiguration 注解，通过 SpringFactoriesLoader 加载 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 中的 130+ 自动配置类。  
  
  每个配置类通过 @ConditionalOnClass、@ConditionalOnMissingBean 等条件注解做守卫——classpath 中有对应依赖、且用户没有自定义 Bean 时，才会生效。  
  
  这就是"自动"的底层逻辑：我帮你配好默认的，但你配的永远优先。  

# 核心注解与 Starter 机制
## 核心注解体系
### 启动层
  ```text
  @SpringBootApplication
      ├── @SpringBootConfiguration    =  @Configuration（标记为配置类）
      ├── @EnableAutoConfiguration    =  开启自动配置（上面讲的核心）
      └── @ComponentScan              =  扫描当前包及子包的 @Component
  ```

### Web 层
  ```text
  @RestController          // = @Controller + @ResponseBody
  @RequestMapping("/api")  // 类级请求路径映射
  
  @GetMapping("/users")     // 简化写法
  @PostMapping
  @PutMapping
  @DeleteMapping
  @PatchMapping
  
  @PathVariable             // URL 路径参数
  @RequestParam             // 查询参数
  @RequestBody              // JSON 请求体绑定
  @RequestHeader            // 请求头绑定
  @RequestAttribute         // 请求属性
  ```

### 依赖注入
  ```text
  @Component            // 通用组件
  @ervice              // 业务逻辑层
  @Repository           // 数据访问层（自动翻译持久层异常）
  @Controller           // Web 控制器层
  
  @Autowired            // 按类型注入（Field/Constructor/Setter）
  @Resource             // JSR-250，按名称注入
  @Inject               // JSR-330，按类型注入
  
  @Qualifier("beanName") // 按名称限定（解决多实现注入歧义）
  @Primary              // 多 Bean 时的首选
  @Scope("prototype")   // 作用域：singleton/prototype/request/session
  @Lazy                 // 懒加载
  ```

  重点：构造器注入是最好的实践：  
  // ✅ 推荐：构造器注入（不可变、便于测试、明确依赖）   
  ```java
  public class UserService {
      private final UserRepository userRepository;
      
      public UserService(UserRepository userRepository) {  // Spring 4.3+ 单构造器无需 @Autowired
          this.userRepository = userRepository;
      }
  }
  ```

  // ❌ 不推荐：字段注入（难以测试、隐藏依赖）
  ```java
  @Service
  public class UserService {
      @Autowired
      private UserRepository userRepository;
  }
  ```

### 配置属性绑定
  ```java
  @ConfigurationProperties(prefix = "app.datasource")
  @Data
  public class DataSourceProperties {
      private String url;
      private String username;
      private String password;
      private int maxPoolSize = 10;  // 默认值
  }
  ```

## Starter 机制
### Starter 是什么
  一个 Maven/Gradle 依赖坐标，把某类功能的所有依赖打包在一起，引入一个 Starter = 引入一堆相关依赖 + 自动生效的自动配置：  
  ```pom
  <!-- 引入这一个 = 引入 spring-boot-starter-data-redis + lettuce + commons-pool2 + ... -->
  <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-data-redis</artifactId>
  </dependency>
  ```

### Starter 的内部结构
  以 spring-boot-starter-data-redis 为例：  
  ```text
  spring-boot-starter-data-redis.jar
  ├── pom.xml              # 只声明依赖，没有代码
  │   ├── spring-data-redis
  │   ├── lettuce-core
  │   └── commons-pool2
  │
  spring-boot-autoconfigure.jar  # 自动配置在这里
  └── META-INF
      └── spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
          └── redis 对应的自动配置类
              └── RedisAutoConfiguration            # 自动配置 RedisTemplate/LettuceConnectionFactory
              └── RedisReactiveAutoConfiguration    # 响应式 Redis
              └── RedisRepositoriesAutoConfiguration # Spring Data Redis Repositories
  ```

  关键理解：Starter jar 本身只做依赖管理（一个空的 pom），真正的自动配置代码在 spring-boot-autoconfigure 这个公共包里。  

### 自定义 Starter 步骤
  假设要为"短信服务"做一个 Starter：  
  ```text
  sms-spring-boot-starter/          # 启动器（空，只管理依赖）
  ├── pom.xml → 依赖 sms-spring-boot-autoconfigure
  └── META-INF/spring.provides      # 可选，描述信息
  
  sms-spring-boot-autoconfigure/    # 自动配置模块（真正的代码）
  ├── META-INF/spring/
  │   └── org.springframework.boot.autoconfigure.AutoConfiguration.imports
  │       └── com.example.sms.SmsAutoConfiguration
  ├── SmsProperties.java            # @ConfigurationProperties(prefix = "sms")
  ├── SmsTemplate.java              # 核心业务类
  └── SmsAutoConfiguration.java     # 自动配置类
  ```

  自动配置类的骨架：  
  ```java
  @Configuration
  @ConditionalOnClass(SmsTemplate.class)
  @EnableConfigurationProperties(SmsProperties.class)
  public class SmsAutoConfiguration {
      
      @Bean
      @ConditionalOnMissingBean  // 用户可覆盖
      public SmsTemplate smsTemplate(SmsProperties properties) {
          return new SmsTemplate(properties);
      }
  }
  ```

  使用者只需：  
  ```pom
  <dependency>
      <groupId>com.example</groupId>
      <artifactId>sms-spring-boot-starter</artifactId>
  </dependency>
  ```

  ```yaml
  sms:
    access-key: xxx
    secret-key: xxx
  ```

# 配置文件管理
## 配置文件的优先级（由高到低）  
  当 Spring Boot 启动时，按以下顺序加载配置，后面的覆盖前面的：  
  1. 命令行参数                   java -jar app.jar --server.port=9090
  2. JNDI 属性                   java:comp/env
  3. Java 系统属性               System.getProperties()
  4. OS 环境变量                 SPRING_DATASOURCE_URL
  5. application-{profile}.yml   (测试环境专用配置)
  6. application-{profile}.yml   (生产环境专用配置)
  7. application.yml             (通用配置，最低优先级)

  重点：application.yml 是最低优先级，命令行参数是最高的。  

## profile 多环境管理
  ```yaml
  # application.yml（公共配置）
  server:
    port: 8080
  
  spring:
    profiles:
      active: dev  # 激活哪个 profile
      
  ---
  # application-dev.yml（开发环境）
  server:
    port: 8081
  
  spring:
    datasource:
      url: jdbc:mysql://localhost:3306/dev_db
      
  ---
  # application-prod.yml（生产环境）
  server:
    port: 8080
  
  spring:
    datasource:
      url: jdbc:mysql://prod-host:3306/prod_db
      hikari:
        maximum-pool-size: 20
  ```

  启动时指定 Profile：  
  ```text
  # 方式一：命令行
  java -jar app.jar --spring.profiles.active=prod
  
  # 方式二：环境变量
  export SPRING_PROFILES_ACTIVE=prod
  
  # 方式三：application.yml 中指定
  spring.profiles.active: prod
  ```

## @Value 与 @ConfigurationProperties
  @Value（简单取值）  
  ```java
  @Component
  public class AppConfig {
      @Value("${app.name:defaultName}")   // 带默认值
      private String appName;
      
      @Value("#{systemProperties['user.home']}")  // SpEL 表达式
      private String userHome;
      
      @Value("${app.thread-pool:10}")    // 类型会自动转换
      private int threadPool;
  }
  ```

  @ConfigurationProperties（结构化绑定，推荐）  
  ```java
  @Data
  @ConfigurationProperties(prefix = "app.datasource")
  @Component  // 或 @EnableConfigurationProperties(AppDatasourceProperties.class)
  public class AppDatasourceProperties {
      private String url;
      private String username;
      private String password;
      private Pool pool = new Pool();   // 嵌套类
      
      @Data
      public static class Pool {
          private int maxSize = 10;
          private int minIdle = 2;
      }
  }
  ```

  对应的 yaml：  
  ```yaml
  app:
    datasource:
      url: jdbc:mysql://localhost:3306/db
      username: root
      password: 123456
      pool:
        max-size: 20
        min-idle: 5
  ```

  区别对比  
  | 特性 | @Value | @ConfigurationProperties |
  |------|--------|--------------------------|
  | 适用场景 | 少量简单属性 | 组属性、嵌套结构 |
  | 类型安全 | 手动转换 | 自动转换 + 校验 |
  | 复杂结构 | 不支持嵌套 | 支持嵌套对象/List/Map |
  | 验证支持 | 无 | 支持 JSR-303（@Validated） |
  | 松散绑定 | 不支持（必须精确匹配） | 支持（如 pool-max-size = pool.maxSize） |

# 内嵌 Tomcat
## 工作原理
  Spring Boot 启动时：  
  ```text
  SpringApplication.run()
    → 推测应用类型（WebApplicationType）
      → SERVLET: 启动内嵌 Tomcat
      → REACTIVE: 启动 Netty（WebFlux）
      → NONE: 非 Web 应用
    → 创建 Tomcat Servlet Web Server Factory
      → TomcatServletWebServerFactory
        → 创建 Tomcat 实例
        → 设置端口（server.port）
        → 配置连接器（Connector）
        → 配置线程池（Executor）
        → 部署 DispatcherServlet
        → 调用 Tomcat.start() 启动
  ```

## 配置内嵌 Tomcat
  ```yaml
  server:
    port: 8080
    servlet:
      context-path: /api          # 所有请求的前缀
      session:
        timeout: 30m              # Session 超时时间
    tomcat:
      max-threads: 200            # 最大工作线程数（默认 200）
      min-spare-threads: 10       # 最小空闲线程数
      max-connections: 10000      # 最大连接数（默认 8192）
      accept-count: 100           # 等待队列大小
      connection-timeout: 5000ms  # 连接超时
      max-swallow-size: 2MB       # 最大请求体吞入大小
      threads:
        max: 200                  # Spring Boot 2.3+ 改用 server.tomcat.threads.max
        min-spare: 10
      basedir: /tmp/tomcat        # Tomcat 临时目录
      accesslog:
        enabled: true             # 开启访问日志
        directory: logs/
        pattern: "%h %l %u %t \"%r\" %s %b"
  ```

## Tomcat 线程模型（面试高频）
  ```text
                     ┌──────────────────────┐
  HTTP Request ──────▶│   Acceptor 线程 (1)   │ 接受 TCP 连接
                      └──────────┬───────────┘
                                 │ 放入
                      ┌──────────▼───────────┐
                      │   Poller 线程 (2)     │ NIO 事件轮询，读取请求
                      └──────────┬───────────┘
                                 │ 提交给
                      ┌──────────▼───────────┐
                      │  Worker 线程池        │ 执行业务逻辑
                      │  (默认 200 个)        │
                      └──────────────────────┘
  ```

  三个线程各司其职：  
  - Acceptor（1个）：只负责 accept 新的 TCP 连接，不处理请求
  - Poller（2个）：NIO 轮询器，读取请求数据、写出响应
  - Worker（默认200）：真正执行 Servlet 代码的业务线程

## 替换为其他容器
  ```pom
  <!-- 排除 Tomcat -->
  <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
      <exclusions>
          <exclusion>
              <groupId>org.springframework.boot</groupId>
              <artifactId>spring-boot-starter-tomcat</artifactId>
          </exclusion>
      </exclusions>
  </dependency>
  
  <!-- 引入 Undertow（更轻量、高并发表现更好） -->
  <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-undertow</artifactId>
  </dependency>
  ```

  ```yaml
  # Undertow 配置
  server:
    undertow:
      threads:
        io: 4                    # IO 线程数（通常 = CPU 核心数）
        worker: 200              # 工作线程数
      direct-buffers: true       # 使用直接缓冲区
  ```

  对比：  
  | 特性 | Tomcat | Undertow | Jetty |
  |------|--------|----------|-------|
  | 默认 | ✅ Spring Boot 默认 | 需手动切换 | 需手动切换 |
  | IO 模型 | NIO | NIO | NIO |
  | 内存占用 | ~30MB 基础 | ~10MB 基础（最轻量） | ~15MB |
  | 高并发 | 好 | 更好（特别适合长连接） | 好 |
  
# Actuator 监控
## Actuator 是什么
  Spring Boot Actuator 提供生产级监控端点，用于查看应用运行状态、健康检查、环境信息、线程栈、指标等。  
  ```pom
  <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-actuator</artifactId>
  </dependency>
  ```

##  核心端点
  ```yaml
  # 暴露所有端点
  management:
    endpoints:
      web:
        exposure:
          include: "*"            # 暴露所有端点
          # include: health,info,env,metrics  # 或按需暴露
    endpoint:
      health:
        show-details: always      # 显示详细健康状态
      shutdown:
        enabled: true             # 启用优雅关闭端点
  ```

  | 端点 | 路径 | 作用 | 默认启用 |
  |------|------|------|----------|
  | health | /actuator/health | 应用健康检查 | ✅ |
  | info | /actuator/info | 应用基本信息 | ✅ |
  | env | /actuator/env | 环境配置属性 | ❌ |
  | metrics | /actuator/metrics | 应用指标（内存/GC/线程/请求） | ❌ |
  | mappings | /actuator/mappings | 所有请求映射 | ❌ |
  | beans | /actuator/beans | 所有 Bean 列表 | ❌ |
  | threaddump | /actuator/threaddump | 线程栈快照 | ❌ |
  | heapdump | /actuator/heapdump | JVM 堆转储 | ❌ |
  | loggers | /actuator/loggers | 日志级别查看/动态修改 | ❌ |
  | shutdown | /actuator/shutdown | 优雅关闭应用 | ❌ |
  
## 自定义健康指标
  ```java
  @Component
  public class DatabaseHealthIndicator implements HealthIndicator {
      @Autowired
      private DataSource dataSource;
      
      @Override
      public Health health() {
          try (Connection conn = dataSource.getConnection()) {
              if (conn.isValid(3)) {  // 3秒超时
                  return Health.up()
                      .withDetail("database", "MySQL")
                      .withDetail("status", "connected")
                      .build();
              }
              return Health.down()
                  .withDetail("error", "连接验证失败")
                  .build();
          } catch (Exception e) {
              return Health.down(e).build();
          }
      }
  }
  ```

  输出：  
  ```json

      "status": "UP",
      "components": {
          "db": {
              "status": "UP",
              "details": {
                  "database": "MySQL",
                  "status": "connected"
              }
          },
          "redis": { "status": "UP" },
          "diskSpace": { 
              "status": "UP", 
              "details": { "free": 8589934592, "threshold": 10485760 }
          }
      }
  }
  ```

## 动态修改日志级别（生产救急）
  ```bash
  # 查看当前日志级别
  GET /actuator/loggers/com.example.service
  
  # 响应
  {
      "configuredLevel": "INFO",
      "effectiveLevel": "INFO"
  }
  
  # 动态改为 DEBUG（这无需重启！）
  POST /actuator/loggers/com.example.service
  Content-Type: application/json
  
  {
      "configuredLevel": "DEBUG"
  }
  ```

  这和 logger.fine()/logger.info() 的面试考点直接挂钩——生产排查问题时，logback 的配置不是只能靠改文件重启，Actuator 端点可以运行时动态调级别，排完再调回去。

## 面试题速答
  | 问题 | 一句话答案 |
  |------|------------|
  | 自动配置怎么实现的？ | @EnableAutoConfiguration → 加载 AutoConfiguration.imports → 130+ 配置类通过 @Conditional 条件守卫判断是否生效 |
  | @ConditionalOnMissingBean 作用？ | 用户自定义了同类 Bean 时，自动配置跳过——保证"用户优先" |
  | Starter 和自动配置什么关系？ | Starter 只做依赖聚合（空的 pom），自动配置代码在 spring-boot-autoconfigure 包里 |
  | 配置文件加载顺序？ | 命令行参数 > 环境变量 > profile 配置 > application.yml |
  | 内嵌 Tomcat 启动过程？ | SpringApplication.run() → 创建 TomcatServletWebServerFactory → 初始化 Tomcat 实例 → 配置 Connector + Executor → 部署 DispatcherServlet → start() |
  | Actuator 有什么用？ | 健康检查(/health)、指标监控(/metrics)、日志级别动态调整(/loggers)、线程栈(/threaddump)、堆转储(/heapdump) |
