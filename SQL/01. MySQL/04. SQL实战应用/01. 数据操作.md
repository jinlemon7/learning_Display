# 数据插入(INSERT)
  INSERT 语句用于向数据表中插入新的行数据。  

## 基本语法
  INSERT 语句有两种基本形式：  
  ```sql
  -- 形式一：指定列名和值
  INSERT INTO 表名 (列1, 列2, ...) VALUES (值1, 值2, ...);
  
  -- 形式二：为所有列指定值
  INSERT INTO 表名 VALUES (值1, 值2, ...);
  ```
  - 使用第二种‌‌‌形式‌时，必须为表中的⁢⁢⁢每一列提供值，并且⁢顺‏‏‏序必须与表定义中的列顺‍‍‍序‏一致。因此，第一⁡⁡⁡种形式通常更安全，也‍更清晰

## 插入单行数据
  假设我们有一个 users 表，包含以下列  
  | 列名               | 数据类型         | 说明                       |  
  |-------------------|----------------|----------------------------|  
  | user_id           | INT            | 主键，自增                 |  
  | username          | VARCHAR(50)    | 用户名，唯一               |  
  | email             | VARCHAR(100)   | 电子邮件，唯一             |  
  | registration_date | DATETIME       | 注册日期                   |  
  | status            | VARCHAR(10)    | 用户状态                   |  

  插入单行数据：  
  ```sql
  INSERT INTO users (username, email, registration_date, status)
  VALUES ('鱼皮', 'yupi@example.com', '2025-08-01 10:30:00', 'active');
  ```
  -  注意，我们没有为 user_id 列指定值，因为它是自增的主键，数据库会自动生成。

## 插入多行数据
  INSERT 语句也可以一次插入多行数据：  
  ```sql
  INSERT INTO users (username, email, registration_date, status)
  VALUES 
      ('代码鸭', 'duck@example.com', '2025-08-02 14:20:00', 'active'),
      ('算法达人', 'algo@example.com', '2025-08-03 09:15:00', 'active'),
      ('面试鸭', 'interview@example.com', '2025-08-04 16:45:00', 'pending');
  ```
  - 这种方式比‌‌‌‌多次执行单行⁢插⁢入⁢更⁢高效，‏特别‏是当‏需要‏‍插入大量‍‍数据时⁡。

## 从其他表插入数据
  INSERT 语句可以结合 SELECT 语句，从其他表中选择数据并插入：  
  ```sql
  INSERT INTO active_users (user_id, username, email)
  SELECT user_id, username, email
  FROM users
  WHERE status = 'active';
  ```
  - 这个语句从 users 表中选择所有状态为 'active' 的用户，并将它们的 ID、用户名和电子邮件插入到 active_users 表中。

## 处理重复记录
  当插入可能导致唯一约束冲突时，可以使用特定的语法来处理。在 MySQL 中，可以使用 ON DUPLICATE KEY UPDATE：  
  ```sql
  INSERT INTO users (user_id, username, email, status)
  VALUES (1, '鱼皮', 'yupi@example.com', 'active')
  ON DUPLICATE KEY UPDATE status = 'active';
  ```
  在 PostgreSQL 中，可以使用 ON CONFLICT：  
  ```sql
  INSERT INTO users (user_id, username, email, status)
  VALUES (1, '鱼皮', 'yupi@example.com', 'active')
  ON CONFLICT (user_id) DO UPDATE SET status = 'active';
  ```

## 获取插入的自增 ID
  在许多情况‌‌‌‌下，需要知道⁢自⁢动⁢生⁢成的主‏‏键值。不‏同的‏‍数据库‍系统有‍不⁡同的‍方⁡法：  
  - MySQL：LAST_INSERT_ID()
  - PostgreSQL：RETURNING id
  - SQL Server：SCOPE_IDENTITY()
  - SQLite：last_insert_rowid()

  <img width="2082" height="1088" alt="image" src="https://github.com/user-attachments/assets/d4541378-32d3-427f-b6ba-0859d829792b" />  

  例如，在 MySQL 中：  
  ```sql
  INSERT INTO users (username, email, status)
  VALUES ('新用户', 'new@example.com', 'active');
  
  SELECT LAST_INSERT_ID() AS user_id;
  ```

  在 PostgreSQL 中：  
  ```sql
  INSERT INTO users (username, email, status)
  VALUES ('新用户', 'new@example.com', 'active')
  RETURNING user_id;
  ```

# 数据更新(UPDATE)
  UPDATE 语句用于修改表中已有的数据。  

## 基本语法
  ```sql
  UPDATE 表名
  SET 列1 = 值1, 列2 = 值2, ...
  WHERE 条件;
  ```
  - WHERE 子句是 UPDATE 语句中最重要的部分，它确定了最终哪些行会被更新。如果省略 WHERE 子句，表中的所有行都会被更新，但这样可能会导致很多问题。

## 更新单行数据
  ```sql
  UPDATE users
  SET status = 'inactive'
  WHERE user_id = 3;
  ```
  - 这个语句将‌‌‌‌ ID 为 ⁢3⁢ ⁢的⁢用户状‏态更‏新为‏ '‏‍ina‍ctiv⁡‍e'‍。

## 更新多行数据
  ```sql
  UPDATE users
  SET status = 'inactive'
  WHERE registration_date < '2025-01-01' AND last_login_date < '2025-06-01';
  ```
  - 这个语句将注‌‌‌‌册日期早于 2025 ⁢⁢⁢⁢年，且最后登录日期早于‏‏‏‏ 2025 年 6 月‍‍‍‍的所有用户的状态更新为⁡⁡⁡⁡ 'inactive'。

## 使用表达式更新数据
  UPDATE 语句中的 SET 子句可以包含表达式：  
  ```sql
  UPDATE products
  SET price = price * 0.9
  WHERE category = '编程书籍';
  ```
  - 这个语句会将所有编程书籍的价格降低 10%。

## 使用子查询更新数据
  UPDATE 语句可以使用子查询来确定要更新的值  
  ```sql
  UPDATE users
  SET status = 'vip'
  WHERE user_id IN (
      SELECT user_id
      FROM orders
      GROUP BY user_id
      HAVING SUM(amount) > 1000
  );
  ```
  - 这个语句将‌所‌‌‌有总消费金额超过⁢ 100⁢⁢⁢0 的用户‏的状态更新为 ‏‏‏'v‍ip'。

## 更新多个表
  在某些数据‌‌‌‌库系统（如 M⁢⁢y⁢S⁢QL）中，‏‏可以‏在一‏个 U‍‍PDA‍TE ‍语⁡⁡句中更新⁡多个表：  
  ```sql
  UPDATE users u
  JOIN orders o ON u.user_id = o.user_id
  SET u.status = 'vip', o.discount = 0.1
  WHERE o.amount > 500;
  ```
  - 这个语句同时更新了 users 表和 orders 表，将大订单（金额超过 500）的用户状态设为 'vip'，并为这些订单添加 10% 的折扣。

## UPDATE 的限制和注意事项
  1. 主键和唯一约束：更新主键或唯一列的值时，需要保证新值不会导致约束冲突
  2. 外键约束：更新被外键引用的列时，需要考虑对相关表的影响。
  3. 触发器：表上的触发器可能会在更新操作时执行，这可能会导致额外的操作或约束。
  4. 大规模更新：对大量数据的更新可能会影响性能，考虑分批更新或在非高峰时段执行。
  5. 审计跟踪：在生产环境中，重要的更新操作应该记录在审计日志中。

# 数据删除(DELETE)
  DELETE 语句用于从表中删除行。  

## 基本语法
  ```sql
  DELETE FROM 表名
  WHERE 条件;
  ```
  - 与 UPDATE ‌‌‌‌一样，WHERE 子句对于 DEL⁢⁢⁢⁢ETE 语句至关重要。如果省略 W‏‏‏‏HERE 子句，表中的所有行都会被‍‍‍‍删除

## 删除单行数
  ```sql
  DELETE FROM users
  WHERE user_id = 5;
  ```
  - 这个语句将删除 ID 为 5 的用户。

## 删除多行数据
  ```sql
  DELETE FROM logs
  WHERE log_date < DATE_SUB(CURRENT_DATE, INTERVAL 30 DAY);
  ```
  - 这个语句将‌‌‌删除 30 天前的⁢⁢⁢所有日志记录。

## 使用子查询删除数据
  DELETE 语句可以使用子查询来确定要删除的行：  
  ```sql
  DELETE FROM cart_items
  WHERE user_id IN (
      SELECT user_id
      FROM carts
      WHERE last_activity < DATE_SUB(CURRENT_DATE, INTERVAL 90 DAY)
  );
  ```
  - 这个语句将‌删除 ‌90 天内没有⁢活动的用户的购物车‏项⁢目。

## 使用连接删除数据
  在某些数据库系统中，可以使用连接来确定要删除的行：  
  ```sql
  -- MySQL 语法
  DELETE users
  FROM users
  JOIN inactive_accounts ON users.user_id = inactive_accounts.user_id;
  
  -- SQL Server 语法
  DELETE u
  FROM users u
  INNER JOIN inactive_accounts ia ON u.user_id = ia.user_id;
  ```
  - 这个语句删除了所有在 inactive_accounts 表中有记录的用户。

## TRUNCATE TABLE 语句
  当需要删除表中的‌‌‌‌所有行时，可以使用 TRUNC⁢⁢⁢⁢ATE TABLE 语句，比 ‏‏‏‏DELETE 更高效：  
  `TRUNCATE TABLE logs;`  
  
  TRUNC‌‌‌‌ATE TAB⁢⁢L⁢E⁢ 与不带 ‏‏WH‏ER‏E 子‍‍句的 ‍DEL‍E⁡⁡TE 的⁡主要区别⁡在于：  
  1. TRUNCATE TABLE 更快，因为它不会逐行删除数据。
  2. TRUNCATE TABLE 会重置自增列的计数器。
  3. TRUNCATE TABLE 不会触发行级触发器。
  4. TRUNCATE TABLE 是一个 DDL 操作，无法在事务中回滚。

  因此，在使‌‌‌‌用 TRUNC⁢⁢A⁢T⁢E TAB‏‏LE‏ 前‏，需要‍‍确保真‍的要删‍除⁡⁡表中的⁡所有数据。  

## DELETE 的限制和注意事项
  1. 外键约束：删除被外键引用的行时，可能会受到约束，除非定义了级联删除。
  2. 触发器：表上的触发器可能会在删除操作时执行。
  3. 大规模删除：大量数据的删除可能会导致性能问题，考虑分批删除或在非高峰时段执行。
  4. 审计跟踪：在生产环境中，重要的删除操作应该记录在审计日志中。
  5. 软删除：在许多应用中，实际上并不物理删除数据，而是使用"软删除"（通过标记行为已删除）。

  <img width="2804" height="414" alt="image" src="https://github.com/user-attachments/assets/0ef22d92-cad9-47e7-832e-16aa453bfabc" />  

## 软删除的实现
  软删除是通过添加一个标记列（如 is_deleted 或 deleted_at）来实现的，而不是实际从表中删除行：  
  ```sql
  -- 物理删除
  DELETE FROM users WHERE user_id = 5;
  
  -- 软删除（更新而非删除）
  UPDATE users SET is_deleted = TRUE WHERE user_id = 5;
  -- 或者
  UPDATE users SET deleted_at = CURRENT_TIMESTAMP WHERE user_id = 5;
  ```

  然后，查询‌‌时‌‌过滤掉已删除的⁢行⁢：  
  ```sql
  SELECT * FROM users WHERE is_deleted = FALSE;
  -- 或者
  SELECT * FROM users WHERE deleted_at IS NULL;
  ```
  - 软删除的优‌‌‌‌点是可以保留历史数据⁢⁢⁢⁢，以及更容易地恢复"‏‏‏‏删除"的数据。缺点是‍‍‍‍查询变得更复杂，并且⁡⁡⁡⁡表的大小会持续增长。

# 事务和数据操作（TRANSACTION、COMMIT、ROLLBACK、SAVEPOINT）
  在执行数据‌‌‌‌操作（特别是多步骤操作⁢⁢）⁢⁢时。事务可以确保‏‏‏‏一组操作要么全部成功‍‍‍‍，要么全部失败，这样⁡⁡⁡⁡就维护了数据的完整性。  

## 基本事务语法
  ```sql
  -- 开始事务
  BEGIN TRANSACTION;  -- 或 START TRANSACTION;
  
  -- 执行数据操作
  INSERT INTO orders (user_id, product_id, amount) VALUES (1, 101, 199);
  UPDATE products SET stock = stock - 1 WHERE product_id = 101;
  UPDATE user_stats SET total_orders = total_orders + 1 WHERE user_id = 1;
  
  -- 如果一切正常，提交事务
  COMMIT;
  
  -- 如果出现问题，回滚事务
  ROLLBACK;
  ```
  - 使用事务可以‌‌‌‌确保这三个操作要么全部⁢⁢⁢⁢成功，要么全部失败。例‏‏‏‏如，如果更新产品库存时‍‍‍‍发现库存不足，可以回滚⁡⁡⁡⁡整个事务，取消订单插入。

## 事务的 ACID 属性
  事务具有四个重要的属性，通常缩写为 ACID：  
  1. 原子性（Atomicity）：事务中的所有操作要么全部执行，要么全部不执行。
  2. 一致性（Consistency）：事务将数据库从一个一致状态转变为另一个一致状态。
  3. 隔离性（Isolation）：同时执行的事务彼此隔离，互不影响。
  4. 持久性（Durability）：一旦事务提交，其结果将永久保存在数据库中。

## 练习 3：数据删除与事务
  编写一个事‌‌‌务，安全地删除以⁢⁢下⁢数据：    ‏‏  ‏      ‍‍   ‍     ⁡⁡    ⁡        
  1. 删除所有标记为"测试"的简历（在 resumes 表中）
  2. 删除这些简历关联的所有技能项（在 resume_skills 表中）
  3. 从用户统计表（user_stats）中更新相关用户的简历计数

  答案：  
  ```sql
  BEGIN TRANSACTION;
  
  -- 获取要删除的简历IDs
  CREATE TEMPORARY TABLE temp_deleted_resumes AS
  SELECT resume_id, user_id FROM resumes WHERE is_test = TRUE;
  
  -- 删除关联的技能项
  DELETE FROM resume_skills 
  WHERE resume_id IN (SELECT resume_id FROM temp_deleted_resumes);
  
  -- 删除测试简历
  DELETE FROM resumes 
  WHERE resume_id IN (SELECT resume_id FROM temp_deleted_resumes);
  
  -- 更新用户统计
  UPDATE user_stats us
  SET resume_count = resume_count - (
      SELECT COUNT(*) 
      FROM temp_deleted_resumes dr 
      WHERE dr.user_id = us.user_id
  )
  WHERE user_id IN (SELECT user_id FROM temp_deleted_resumes);
  
  -- 清理临时表
  DROP TEMPORARY TABLE IF EXISTS temp_deleted_resumes;
  
  COMMIT;
  ```
