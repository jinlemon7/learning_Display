# 第一轮：关于统一网关与认证
  面试官：你说用了 Spring Cloud Gateway 做统一入口，并用 JWT 做全链路鉴权。我有三个问题：  

## 问题一：你的 JWT 里存了什么？如果把用户角色、权限都存进去，Token 会不会太大？怎么权衡？  

   (JWT设计)：JWT里只存最核心的、高频使用的、不敏感的声明，比如 userId, userName。绝不把角色、权限明细这类可能变化或过重的信息塞进去，避免HTTP Header过大。权限校验的逻辑在网关层或下游服务里，根据userId去Redis或本地缓存里查。  

## 问题二：你提到了“防重放”机制。具体是怎么实现的？如果用户网络卡顿，连续点了两下提交，你的“防重放”机制会误伤正常请求吗？你的方案和标准的“幂等性”设计有什么区别？

  (防重放与幂等)：防重放主要防范恶意窃取Token后重复使用，核心是在JWT里加入时间戳iat、过期时间exp和一个绝对唯一的随机数jti (JWT ID)。服务端用Redis记录处理过的jti，有效期设为Token本身的有效期。这不会误伤正常请求。用户快速双击提交的是业务层请求，不是同一个Token。我们真正的应对方案是业务幂等性设计：在支付、下单等关键接口，前端先向后端申请一个全局唯一的idempotency-key，后端以此键在Redis中做防重，从而区分“协议层攻击”和“业务层重试”。  

## 问题三：你们做了鉴权，那“鉴权”和“授权”的区别是什么？在一个微服务“全链路”的调用里，下游服务比如订单服务收到请求时，它还需要再做一次权限校验吗？为什么？
  (鉴权vs授权)：鉴权是“证明你是谁”（验证JWT签名），授权是“你能干什么”（校检权限）。必须做。绝不能盲目信任上游传来的请求。订单服务必须从JWT中取出userId，校验这笔订单的归属，防止“水平越权”（用户A修改用户B的订单）。这是“纵深防御”原则  

# 第二轮：关于微服务架构与核心业务
  面试官：你按业务拆分了多个服务，听起来比较标准。我们现在来聊聊数据一致性这个真正头疼的问题。  

## 问题一：你的“支付服务”和“订单服务”是分离的。那么，当用户支付成功后，支付服务需要通知订单服务将状态改为“已支付”。你如何保证支付状态的变更，一定能最终且正确地同步到订单服务？如果通知失败了呢？

  (支付状态同步)：这是典型的分布式事务问题。同步调用失败概率高，不可取。我们的方案是基于消息队列的最终一致性。  
  1. 本地事务 + 消息表：支付服务处理完支付流水，在其本地数据库中，将“支付流水”与“待发送的通知消息”写入同一个事务。
  2. 定时任务兜底：一个后台Job轮询未成功发送的消息，进行补发。
  3. 订单服务幂等处理：订单服务用流水号保证只处理一次。

## 问题二：你设计了订单状态机来支持“争议裁决”。请说清楚，一次完整的“用户下单 -> 支付 -> 卖家执行 -> 验收 -> 最终支付”流程里，状态机有多少个状态？触发状态变迁的核心事件有哪些？如果用户长时间不“验收”，系统该怎么处理？如果“争议”期间，这笔托管资金的管理策略是怎样的？

  (状态机设计)：  
  - 状态：待支付 -> 已支付(托管中) -> 执行中 -> 待验收 -> 已完成 / 争议中 -> 退款中 -> 已关闭。
  - 核心事件：用户支付、CEO接单、CEO提交执行完成、用户确认验收、用户提起争议、平台裁决、系统超时自动完成。
  - 超时验收处理：设置验收超时（如7天），由定时任务扫描，触发超时事件，将状态自动流转，并释放托管资金给卖家。
  - 资金策略：争议期间资金严格托管在支付机构的“中间户”，直到平台裁决结果，资金永远不能由平台直接打入个人账户。

## 你说到“支付服务统一接入支付宝/微信/银行渠道”。这是个典型的策略模式应用场景，请具体描述一下你的代码层面或者架构上是如何实现这种多渠道路由和切换的？

# 关于数据、缓存与高并发
  面试官：现在我们谈谈数据和缓存，这是系统并发能力的根本。  

## 问题一：你说Redis缓存了“订单与推荐结果”。订单作为动态变化的实体，你是怎么设计它的缓存方案的？是直接把整个订单对象序列化进去吗？当订单状态从“待支付”变为“已支付”时，你是选择更新缓存中的数据，还是直接删除缓存？为什么？
  (订单缓存策略)：延迟双删 + 最终一致性。不适合存整个对象。  
  1. 缓存标记法：对每个用户缓存近期订单ID列表。查询时先查ID列表，再根据ID单个或批量从主存/缓存获取简要信息。
  2. 更新策略：采用先删除缓存的策略。因为缓存中的数据是派生出来的，直接删掉，等下次请求再重建，比“更新一个可能出错、并发竞争复杂的字段”要简单安全得多。


## 问题二：你提到“MySQL表结构最小化设计”并且用ENUM保证状态一致性。ENUM真的能保证状态一致性吗？它有什么可能被喷的坑？如果要查订单的执行效率，order_status 索引是个字符串索引，在高并发下性能如何？有没有更好的设计？
  (ENUM与状态设计)：ENUM天然防脏数据，但修改成本高（DDL变更）。可以用TINYINT代替ENUM，在代码逻辑中严格保证。  
  - 索引优化：不建议直接用字符串做索引。用TINYINT，配合UNSIGNED，索引效率和内存占用远优于VARCHAR。这是数据库设计的基本优化之道。

## 问题三：假设现在“事务所推荐”这个功能要做一个每分钟刷新的高并发榜单。用Redis的Sorted Set来查排名很合适，那怎么保证缓存和数据库里的事务所分数（比如根据成交量）保持基本一致？能不能直接在Redis里排行，然后定时把结果同步回MySQL？这两种“缓存维护策略”有什么区别？

   (排行榜与缓存一致性)：  
   - 两种策略选择：  
     - Cache-Aside/Sync：分值数据主源头仍在MySQL，Redis只作为查询缓存，通过定时任务同步“增量数据”来更新分值。高并发下直接读排序好的ZSET。
     - Event Sourcing：分值变化事件 -> 消息队列 -> 排序服务消费，先更新MySQL分数值，再同步指令到Redis。
   - 核心对比：能在Redis里排行，然后用BGSAVE或Dump文件恢复数据，但不建议作为唯一数据源。MySQL是持久化、一致性的基石。Redis只是加速查询用的“工艺品”。

# 支付状态同步的消息队列是什么，如何保证最终一致性？如果失败或者服务死机了呢
## 消息队列选型：RocketMQ  
  我选择 RocketMQ，而不是RabbitMQ或Kafka。原因很直接：  
  - 这是一个金融交易场景，要求消息绝对不能丢，顺序不能乱。
  - RocketMQ 支持事务消息，这是保证支付状态和消息发送原子性的关键能力，RabbitMQ和Kafka在这个场景下不如它合适

## 保证最终一致性的核心流程：事务消息 + 本地消息表兜底
  整个流程分为三个角色：支付服务（生产者）、RocketMQ Broker、订单服务（消费者）。
  - 第一步：发送预备消息
     支付服务处理完用户支付后，先不直接提交本地事务，而是向RocketMQ发送一条Half消息（预备消息）。这条消息此时对消费者不可见。
  - 执行本地事务
    RocketMQ收到Half消息后，回调支付服务的本地事务逻辑，也就是写支付流水、扣减余额这些操作。这一步需要在一个本地数据库事务里完成。
    本地消息表：  
    ```java
    支付流水的状态字段设为：status = 'PAID'，notify_status = 'PENDING'
    ```
    注意，这里多了一个 notify_status 字段，用来标记这条支付记录是否已成功通知到订单服务。

  3. 第三步：提交或回滚
     - 如果本地事务提交成功，支付服务通知RocketMQ Commit，消息变为可见，进入死信队列前的兜底。
     - 如果本地事务回滚，支付服务通知RocketMQ Rollback，消息直接丢弃。
    
  4. 消费者处理
     订单服务拉取到消息后，执行业务逻辑：更新订单状态为“已支付”，并在自己的数据库本地事务中落库。处理成功后返回 CONSUME_SUCCESS，消息被消费。

## 四个关键异常场景的兜底方案  
  - 场景一：支付服务本地事务提交成功，但向RocketMQ发送Commit时网络超时  
    RocketMQ的兜底：Broker会定期回查。它发现这条Half消息长时间没有Commit/Rollback指令，会主动调用支付服务提供的回查接口。  
  - 场景二：消息已Commit，但订单服务消费失败（如数据库死锁、业务逻辑异常）  
    - 消费端重试：RocketMQ默认会重试16次，每次间隔递增（10s、30s、1min、2min...）。重试都失败后，消息进入死信队列。
    - 死信队列告警：我们配置了死信队列的监控，一旦有消息进入，立刻钉钉/企微告警。运维介入手工排查消费失败原因，修复后从死信队列重新消费。
  - 场景三：支付服务突然宕机/服务死机
    这个问题分两层处理：
    - 第一层，RocketMQ回查机制：即使服务宕机后重启，Broker的回查不会丢失。它会持续追问，直到拿到明确的Commit或Rollback。Broker的回查是无状态的，依赖支付服务的数据库询问结果。
    - 第二层，本地消息表的定时补偿：即使RocketMQ的回查机制失效（极端情况），我们还有最后一道保险。
      支付服务有一个独立的定时任务，每隔5分钟扫描支付流水表：
      ```java
      SELECT * FROM pay_flow 
      WHERE status = 'PAID' 
      AND notify_status = 'PENDING' 
      AND create_time > DATE_SUB(NOW(), INTERVAL 1 HOUR)
      ```
      扫描出来的记录，重新发送MQ消息。notify_status 在消息发送成功后更新为 'SENT'。
  - 场景四：订单服务消费成功并返回了CONSUME_SUCCESS，但RocketMQ没收到确认（网络闪断）
    这时消息会被重复投递。
    订单服务的幂等处理是关键：订单表有一个 pay_no 唯一字段，或单独建一张 mq_message_log 表记录处理过的 msgId。
    ```java
    // 消费幂等逻辑
    public void handlePayCallback(PayCallBackEvent event) {
        String payNo = event.getPayNo();
        // 数据库唯一索引保障
        Order order = orderService.getByPayNo(payNo);
        if (order != null && "PAID".equals(order.getStatus())) {
            return; // 已处理，直接丢弃
        }
        // 处理业务...
    }
    ```
    消费者端在处理完成后，要返回 CONSUME_SUCCESS。RocketMQ依赖这个返回值确认消费成功。如果没有收到，消息会被重新投递，结合幂等性保证不会重复处理。

### 总结
  第一层：RocketMQ事务消息（90%场景处理）
  第二层：Broker回查机制（覆盖网络超时）
  第三层：本地消息表 + 定时任务补偿（应对极端情况）

# 有什么熔断降级机制
  熔断和降级是防止级联故障的最后一道防线。我的项目里，这套机制主要依托于 Sentinel，并配合网关层做了立体化设计。  

  下面我从触发时机、具体策略、以及落地方案三个层面来阐述。  

## 熔断与降级的总设计思路
  我把整个系统的资源分为三个层次来保护：  
  - 网关层降级：在流量入口就挡掉不健康的服务，返回兜底响应。
  - 服务间熔断：微服务调用链中，下游服务变慢或出错，上游主动切断调用。
  - 业务降级：对非核心功能做优雅取舍，优先保证交易链路的可用。

## 核心熔断策略：基于 Sentinel 的兜底
  我选用 Sentinel 而非 Hystrix，原因有三：一是它支持更细粒度的流控和熔断规则，能实时查看QPS、RT；二是可以动态调整规则，无需重启服务；三是它天然整合 Nacos，规则可以持久化到我正在用的配置中心。  

  具体熔断规则我配置了三种：  
  1. 慢调用比例熔断
     场景：比如支付服务调用银行渠道路由时，平均RT超过阈值（如1秒），说明下游银行接口或网络严重拖慢。
     规则设置：
     - 最大RT：1000ms
     - 比例阈值：50%以上的请求响应超时
     - 熔断时长：10秒
     - 最小请求数：5
     熔断动作：一旦触发，下个时间窗口内的支付请求直接调用失败，抛出 DegradeException，由我们预定义的 Fallback 逻辑接管。
  2. 异常比例/异常数熔断
     场景：订单服务调用推荐服务时，如果推荐服务频繁抛出异常（比如Redis挂了），继续调用只会浪费资源。
     规则设置：
     - 异常比例阈值：50%
     - 统计时长：1分钟
     - 熔断时长：10秒
     熔断动作：快速失败，返回静态的兜底推荐列表（可能是一份热门的差事推荐，缓存在本地），而不至于整个请求被拖垮。
  3. 资源耗尽时的快速失败
     场景：比如支付服务的线程池被打满，或者数据库连接池耗尽。
     规则设置：
     - 基于信号量隔离，限定最大并发线程数
     - 线程数达到阈值时，新请求直接拒绝
     熔断动作：快速返回错误码，避免大量请求堆积，导致服务雪崩。

## 服务降级：多层次兜底
  降级跟熔断互为表里。熔断是短路，降级是短路后“给你什么替代方案”。  

### 层级一：网关层降级  
  在 Spring Cloud Gateway 中，我为每个核心路由集成了 Sentinel 的 SentinelGatewayFilter。  
  场景：如果整个差事列表服务因大促流量冲击而宕机，  
  降级后的响应：  
  ```java
  {
    "code": 2000,
    "message": "系统繁忙，请稍后再试",
    "data": null
  }
  ```
  同时前端会展示“服务繁忙中，请稍后刷新”的提示，而不是一直转圈或崩溃。  

### 层级二：服务层降级
  用 @SentinelResource 注解配合 blockHandler 和 fallback 方法。  
  场景举例：  
  - 推荐服务熔断后，fallback 方法返回一份缓存的“热门差事”兜底列表，保证用户看到的不会是空白页。
  - 支付服务调用失败后，blockHandler 返回明确的错误码给上层，让订单服务将订单置为“待重试”，并提示用户稍后支付。
  - 订单查询高峰期，根据 blockHandler 触发降级，取消不必要的实时聚合计算，只返回主单信息，隐藏执行细节。

### 层级三：依赖隔离与线程池保护
  每个微服务我会用独立的线程池调用下游依赖。  
  例子：支付服务的调用链路里，我会把“调用微信支付”、“调用支付宝”、“记录审计日志”放到三个不同的线程池。即使记录审计日志的线程池满了，支付核心流程不受影响。  

## 降级后的恢复机制
  这是很多人容易忽略的一点。  
  
  自动恢复：Sentinel 熔断器打开后，会经过我预设的“熔断时长”（比如10秒），然后自动切换到半开状态。它允许放行一个请求去试探下游。  

  - 如果这个探测请求成功了，关闭熔断器，恢复正常。
  - 如果探测请求还是失败，重新打开熔断器，进入下一个漫长的恢复窗口。

  预警与人工介入：  
  当熔断器状态发生变化时，Sentinel 会生成日志。我的项目里集成了日志的监控，一旦检测到高频率的熔断打开事件，就会通过钉钉机器人告警，通知运维介入排查根本问题——比如是不是数据库索引没命中，或者第三方的接口出故障了。  

## 总结
  我个人的经验总结是三个词：快速失败、有损服务、立体防护。  

  1. 绝不能让一个请求卡住整个集群，快速失败 > 优雅地等死。
  2. 宁可有损体验，不能服务崩溃。给一堆静态推荐，也比让用户盯着白屏好。
  3. 从上到下都有防线。网关挡不住，服务熔断来挡；服务熔断漏了，业务降级兜底。最后还有监控告警这条底线。

  这就是我项目中完整的熔断降级设计。
