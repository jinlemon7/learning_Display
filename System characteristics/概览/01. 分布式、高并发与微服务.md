| 概念 | 核心范畴 | 关注的问题 | 比喻（一家餐厅） |  
| :--- | :--- | :--- | :--- |  
| **高并发** | 性能/负载需求 | 如何同时服务大量顾客，而不让任何人等太久？ | **客流大**：午饭时间突然涌入1000名顾客，你的出餐系统能不能扛住。 |  
| **分布式** | 物理/部署拓扑 | 如何把工作分给多个厨师、多个厨房协同完成？如果某个厨师病了怎么办？ | **多厨房协作**：后厨不是一人单干，而是有切菜、炒菜、摆盘等多个工作台，靠喊话或传票协作。这解决了单个人手脚慢的问题。 |  
| **微服务** | 架构/组织设计 | 如何按业务拆分团队，让凉菜组、热菜组、甜点组独立运营、互不拖累？ | **专业化独立摊位**：把餐厅拆成“凉菜部”“甜品站”“烧烤铺”，各自独立核算，能独自改良菜品，还能分别根据客流量独立扩招人手。 |  

# 分布式 — 系统的“物理拓扑”与部署范畴
  分布式描述的是系统的物理部署形态，核心是：一个系统的不同组件运行在通过网络相连的多台独立计算机上，彼此通过消息传递来协调行动。  

  - 所属范畴：系统部署与协作方式，解决单机算力有限、单点故障的问题。
  - 核心要解决的事：网络通信、数据一致性、容错、服务发现、CAP 理论等
  - 本质：它并不是一种固定的架构，而是一种底层特性。你的系统可能是分布式的，但不一定就是微服务（比如一个分布式数据库集群、一个 HDFS 集群，它们都是分布式系统，但不是微服务）。

#  高并发 — 系统的“性能需求”与负载特性范畴
  高并发描述的是系统面临的负载特征，指系统在同一时间段内能够同时处理大量请求或承担大量用户操作  

  - 所属范畴：非功能性需求与性能目标，它是一个结果导向的指标，而不是某一种具体的架构。
  - 核心要解决的事：如何提升吞吐量、降低响应延迟、避免共享资源竞争（锁）、防止数据库击穿等。
  - 本质：高并发是许多业务面临的“难题”。解决它可选的手段很多：可以是单机内的多线程、异步化、协程，也可以横向扩展用分布式集群，甚至两者结合。并不是只有分布式架构才能搞定高并发，但高并发的极致场景几乎必然需要分布式。

# 微服务 — 系统的“架构风格”与业务拆分范畴
  微服务是一种架构设计风格，关注如何根据业务边界将单一应用拆分成一组小型的独立服务。  

  - 所属范畴：应用架构与团队组织模式，解决的是“单体应用”在业务复杂、团队膨胀时难以维护、部署和扩展的问题。
  - 核心要解决的事：服务如何拆分、跨服务通信（RPC/消息）、服务治理（注册发现、限流熔断）、数据最终一致性、独立部署等。
  - 本质：微服务首先是面向业务的纵向切分，它天然预设了一个分布式环境。但反过来，分布式系统不一定是微服务（还可以是传统 SOA、大规模并行计算等）。

# 三者如何关联
  - 高并发倒逼分布式：当流量大到单机扛不住时，系统在物理上不得不拆分到多台机器，于是变成了分布式系统。
  - 分布式支撑微服务：微服务将业务拆分成一个个小服务，这些小服务必然运行在不同节点上，所以微服务天然是分布式系统。它从业务角度，为“为什么要把系统拆成这样”提供了方法论。
  - 微服务使分布式治理更复杂，但也更有序：分布式系统带来了网络延迟、数据不一致等固有问题。微服务通过明确的服务边界、独立的数据库等强约朿，把这些分布式问题控制在服务边界内部，并配套服务网格、网关等治理手段。
