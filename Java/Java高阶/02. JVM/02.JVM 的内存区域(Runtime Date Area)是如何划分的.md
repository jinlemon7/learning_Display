内存区域其实指的就是 JVM 的运行时数据区。它分成五个主要区域：元空间、堆、虚拟机栈、本地方法栈和程序计数器。  

这些区域有的线程共享，有的线程私有，帮助 JVM 高效地存储数据、执行代码。如果你想象 JVM 像个大仓库，这些就是不同的“储物间”，各司其职，避免数据乱套。

# 堆（Heap）
  最大的共享区，专门放对象、数组和字符串常量池（全局唯一）。垃圾回收（GC）主要在这儿干活，新对象出生，老对象养老。  

# 堆内存的进一步划分
  - Eden 区：新对象最初会被分配到 Eden 区，且 Eden 区较大，频繁进行垃圾回收。
  - Survivor 区：两个 Survivor 区 S0 和 S1 交替使用，新对象在 Eden 区经过一次垃圾回收后存放到其中一个 Survivor 区，进一步存活的对象会移动到另一个 Survivor 区，最终晋升到老年代。
  - 老年代：长生命周期对象经过多次垃圾回收后会被移到老年代，Major GC 在老年代进行，频率较低但耗时较长。

## 不同的GC方式
  | 对比维度 | Minor GC (Young GC) | Major GC (Old GC) | Full GC |  
  | :--- | :--- | :--- | :--- |  
  | 清理区域 | 仅 Eden + Survivor 区 | **仅老年代** (Tenured / Old Gen) | 整个堆 + 元空间 (Metaspace) |  
  |*触发时机 | Eden 区满了 | 老年代剩余空间不足以容纳晋升对象 | 1. System.gc() 显式调用</br> 2. CMS/Parallel 并发失败</br> 3. 元空间满了 |  
  | 停顿时间 | 较短（毫秒级，取决于存活对象多少） | 较长（通常比 Minor 长） | 很长（秒级甚至分钟级，与堆大小成正比） |  
  | 常见误解 | 无争议 | 注意：很多工具把 Full GC 记录为 Major GC | 这才是真正影响吞吐量的元凶 |  

## 主流GC收集器核心特性速览
  | 特性维度 | Serial GC | Parallel GC | CMS (已废弃) | G1 GC | ZGC / Shenandoah |  
  | :--- | :--- | :--- | :--- | :--- | :--- |  
  | 设计目标 | 简单高效，单线程 | 最大化吞吐量 | 最小化停顿时间 | 可预测的停顿时间 | 极低延迟 (<10ms)，支持大堆 |  
  | 工作方式 | 单线程，STW | 多线程，STW | 并发标记 & 清除 | 并发标记 & 分区混合回收 | 几乎全程并发，极短STW |  
  | 内存布局 | 分代 (Young / Old) | 分代 (Young / Old) | 分代 (Young / Old) | 分区 (Region) | 分区 (Region) |  
  | 回收算法 | 复制 / 标记-整理 | 复制 / 标记-整理 | 标记-清除 | 标记-复制 (局部压缩) | 标记-复制 (并发压缩) |  
  | 碎片处理 | 无碎片 | 无碎片 | **严重碎片问题** | 无碎片 | 无碎片 |  
  | 典型停顿 | 亚秒级~秒级 (与大堆正比) | 亚秒级~秒级 (与大堆正比) | 毫秒级~秒级 (并发失败时) | **~200ms (可配置)** | **1-10ms (与大堆解耦)** |  
  | 吞吐量 | 低 | **高** | 中等 | 中等 | 中等 |  
  | 适用场景 | 客户端小程序，< 100MB堆 | 批处理、科学计算，对停顿不敏感 | **已废弃**，不推荐新项目使用 | 服务端应用，兼顾吞吐与延迟，**JDK 9+默认** | 低延迟要求极高的系统，超大内存 (>16GB) |  

## ZGC和Shenandoah的核心技术
  | 对比维度 | 染色指针 (Colored Pointers) for ZGC | 转发指针 (Forwarding Pointer) for Shenandoah |  
  | :--- | :--- | :--- |  
  | 核心原理 | **信息即指针**：元数据编码在指针位中。 | **引用即间接**：通过一个独立的指针字段转发。 |  
  | 数据位置 | 存储在64位**指针的高位**中。 | 存储在**对象头**的一个额外字段中。 |  
  | 内存开销 | 较低，约**1-2%**。 | 早期较高（约6-10%），后续版本通过复用`mark word`已大幅优化。 |  
  | 访问开销 | 有**自愈能力**，仅在首次访问旧指针时开销较大。 | 每次访问均需通过转发指针，存在**持续的间接寻址开销**。 |  
  | 平台支持 | **仅支持64位平台**（Linux/x64, macOS, AArch64等）。 | **支持所有OpenJDK支持的平台**（包括32位）。 |  

  染色指针技术原理：  
  ZGC利用64位指针未用于寻址的高位（通常是第42-45位，共4位）来存储GC元数据。这些元数据（即“颜色”）用于表示对象状态，例如：  
  - Marked0 / Marked1: 用于并发标记阶段的标记状态。
  - Remapped: 标识对象是否已被迁移，当前引用是否指向最新地址。
  - Finalizable: 标识该对象是否只能通过finalize方法访问。

  GC或应用线程只需解析指针的颜色位，就能立即知晓对象状态，无需访问对象头。  

  转发指针技术原理：  
  Shenandoah在每个对象的对象头（mark word）之前添加一个额外的指针字段。它有两种状态：  
  - 指向自己：对象未被移动。
  - 指向新副本：对象已被移动到新位置，成为一个“转发指针”。

# 元空间和方法区的区别：
  - JDK 7 及之前，HotSpot 使用永久代（PermGen）实现方法区，主要存储类信息（类的结构信息）、常量（运行时常量池，每个类1个）、静态变量（只依赖类、不依赖对象实例的变量）和JIT编译后的机器码等。
  - JDK 8 之后，永久代（方法区）被移除，改为使用元空间（Metaspace），元空间使用本地内存（Direct Native Memory）来提高性能和避免 OOM 错误。元空间可以动态调整大小，而永久代大小是固定的。

# 方法区（Method Area）
  这是大家共享的“知识库”（线程共享区域），存着类的结构信息、常量（运行时常量池，每个类1个）、静态变量（只依赖类、不依赖对象实例的变量）和JIT编译后的机器码。所有线程都能访问。  

  JDK 8 之前用永久代（PermGen）实现，容易内存溢出，现在换成元空间（Metaspace），用本地内存（直接本地内存），更靠谱。  

# 直接本地内存（Direct Native Memory）
  属于 JVM 之外的内存区域，但挺重要。它通过 NIO 的 ByteBuffer 直接分配本机内存，不走堆，受系统内存限制，但速度快，适合大文件传输。  

# 虚拟机栈（VM Stack）
  每个线程独享的工作台，存局部变量表、操作数栈、动态链接和方法出口信息。基本类型变量和对象引用（常量池引用）都存这。  

  注意，栈是线程私有的，生命周期与线程相同。  

# 虚拟机栈和栈帧
  每当一个方法被调用时，虚拟机会在栈中创建一个新的栈帧（Stack Frame），该栈帧用于存储方法的局部变量表、操作数栈、常量池引用等。  
  方法执行完毕后，栈帧会被弹出，释放内存。  

# 本地方法栈（Native Method Stack）
  类似虚拟机栈，但专为 JNI 调用本地代码（比如 C++）服务。  

  也是线程私有的，用来分配内存给那些非 Java 方法。  

# 程序计数器（Program Counter Register）
  每个线程都有一个独立的程序计数器，属于线程私有。作用是保存当前线程执行的字节码指令的地址或行号。  

  线程切换时，它帮你记住位置，不会迷路。唯一不会 OOM 的地方，因为太简单了。
